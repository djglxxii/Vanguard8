# Milestone 45 — Timed HD64180 `AND r` / `AND (HL)` Coverage for PacManV8 T021

Status note, 2026-04-23:
- Milestone 44 was authorized as the single remaining T021 compatibility
  closure milestone and cleared the then-current timed opcode gaps
  (`SUB E`, same-path `SUB C`, follow-on `CPL`). `M44-T01` was moved to
  `docs/tasks/blocked/` when the replay path exposed a PacManV8-side
  fidelity issue (score/dots `0/0`, Pac-Man tile `(14, 26)`) that was
  explicitly outside M44's allowed Vanguard8 scope.
- The PacManV8 side has since resolved that fidelity issue (local
  movement bug in `movement_try_turn_at_center` fixed) and the
  blocked-task log at
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T021-pattern-replay-and-fidelity-testing.md`
  now pins a new Vanguard8-side timed-opcode gap:

  ```text
  terminate called after throwing an instance of 'std::runtime_error'
    what():  Unsupported timed Z180 opcode 0xA3 at PC 0x125B
  ```

  This is `AND E` (`0xA3`) in `collision_consume_tile`, used to mask
  the pellet bit under Pac-Man's current tile:

  ```text
  1256 3A 12 82   ld a, (COLLISION_WORK_MASK)
  1259 5F         ld e, a
  125A 7E         ld a, (hl)
  125B A3         and e
  125C 28 F0      jr z, .none
  125E 7B         ld a, e
  125F 2F         cpl
  1260 A6         and (hl)
  ```

- The same blocked-task log identifies `AND (HL)` (`0xA6`) at
  `PC=0x1260` as the strongly suspected immediate follow-on trap in the
  same pellet-consumption success path.
- Per the M44 philosophy ("replace the one-blocker-per-milestone
  pattern") and the user's direction to cover all identified and
  strongly suspected missing opcodes in a single milestone, this
  milestone bundles the full `AND r` register family together with
  `AND (HL)`. The timed dispatch already includes `AND n` (`0xE6`) and
  `AND A`-style shortcuts used elsewhere; the register-bank and memory
  forms are the missing mirror.

Objective:
- Close the current confirmed timed-opcode gap (`AND E`, `0xA3`) and
  the immediate same-path look-ahead (`AND (HL)`, `0xA6`), and in the
  same pass cover the remaining `AND r` register forms of the same
  family so the PacManV8 T021 replay path cannot abort again on a
  sister `AND r` instruction inside the same rebuild cycle.

Allowed scope:
- Timed opcode coverage in the extracted HD64180 runtime for the
  following base opcodes:
  - `AND B` (`0xA0`)
  - `AND C` (`0xA1`)
  - `AND D` (`0xA2`)
  - `AND E` (`0xA3`) — currently confirmed blocker
  - `AND H` (`0xA4`)
  - `AND L` (`0xA5`)
  - `AND (HL)` (`0xA6`) — currently suspected next blocker
  - `AND A` (`0xA7`)
- Focused CPU coverage for every newly covered opcode (operand/result,
  flag behavior — Z, S, H=1, N=0, C=0, P/V as parity —, `PC` advance,
  and adapter T-state classification; `AND (HL)` also pins the memory
  read).
- Narrow non-perturbation fixture regression proof that existing
  fixture ROMs which do not exercise these opcodes remain byte-
  identical in frame/audio/event-log digests.
- Doc/task updates required to lock the new behavior.

Explicit non-goals:
- **No general CPU opcode sweep.** Only the `AND r` register family
  and `AND (HL)` are in scope. Do not add `OR r`, `XOR r`, `CP r`,
  rotate/shift families, index-register (`IX`/`IY`) forms, or
  ED-prefix forms unless a new T021 repro proves they are required —
  and in that case, open the next milestone, do not broaden this one.
- **No PacManV8 source edits from this repo.** PacManV8 ROM, harness,
  and evidence changes stay in the PacManV8 repo.
- **No unrelated subsystem work.** No VDP, audio, scheduler, debugger,
  save-state, desktop GUI, or broad frontend changes. No headless
  replay/inspection/reporting format changes unless a new T021 repro
  proves one is strictly required, in which case defer to a follow-up
  milestone.
- **No loosening of existing tests.** Do not relax the fail-fast
  `Unsupported timed Z180 opcode` contract for any opcode outside
  this milestone's allowed scope.
- **No undocumented behavior invention.** The `AND` family follows
  published HD64180/Z80 semantics. Implement exactly those semantics.

Required tests:
- Focused timed CPU coverage in the existing CPU test file for each
  newly covered opcode, pinning:
  - Operand/result behavior (A ← A AND operand).
  - Flag behavior: S from bit 7 of result, Z from result, H=1, P/V
    from parity of result, N=0, C=0.
  - `PC` advance (1 byte for `AND r`, 1 byte for `AND (HL)`; no
    operand bytes).
  - Adapter T-state classification (`AND r` base, `AND (HL)` memory
    read form).
- `AND (HL)` test must exercise the memory read so the adapter's
  memory-read timing path is pinned, not only the register path.
- Negative guard preserving the existing fail-fast contract for a
  still-out-of-scope sister opcode (for example `OR E` / `0xB3` or
  `XOR E` / `0xAB`) so an accidental broadening is caught.
- Non-perturbation guard against an existing fixture ROM that does
  not exercise the newly covered behavior: frame, audio, and
  event-log digests must stay byte-identical before and after the
  change.
- Kickoff runtime coverage: the canonical PacManV8 T021 harness must
  advance past both `PC=0x125B` (`AND E`) and `PC=0x1260`
  (`AND (HL)`) without aborting. Either the harness completes
  end-to-end, or it surfaces the next distinct blocker, which must
  then be recorded in the matching task file.

Exit criteria:
- Every opcode listed in Allowed scope is covered by the timed
  extracted core and by a dedicated focused test.
- `ctest --test-dir cmake-build-debug --output-on-failure` passes
  with the new coverage included, and no pre-existing regression is
  relaxed.
- Deterministic digests for unaffected fixture ROMs remain unchanged
  unless a deliberate change is explicitly recorded in the task file.
- The canonical PacManV8 T021 `tools/pattern_replay_tests.py`
  command either completes end-to-end against the current Vanguard8
  tree or stops on a new distinct blocker that is explicitly out of
  M45 scope, which is then recorded in `M45-T01` for the next
  milestone contract.
- The matching task file (`M45-T01`) captures the full opcode list,
  per-opcode test references, verification evidence, and — if a new
  out-of-scope blocker is surfaced — the precise next repro.

Forbidden extras:
- Turning this milestone into a broad extracted-core CPU completion
  pass by including `OR r`, `XOR r`, `CP r`, or other sister families
  without a T021 repro.
- Bundling unrelated real-ROM compatibility work not exercised by the
  PacManV8 T021 replay path.
- Editing PacManV8 sources to route around a Vanguard8 defect.
- Reintroducing the per-opcode milestone split for T021; a single
  follow-up milestone contract is required if a new out-of-scope gap
  is surfaced, not one per opcode.
