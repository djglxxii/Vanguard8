# Milestone 33 - Instruction-Granular Audio Timing for YM2151 Busy Polls

Objective:
- Unblock milestone 32 by fixing the core scheduling defect that lets ROM code
  execute many scheduled CPU instructions without advancing audio hardware time
  between those instructions. The current PacManV8 T017 ROM remains in the
  YM2151 busy-poll/write helper (`audio_ym_write_bc`, `PC 0x2B8B`) through at
  least 3,600 frames. VDP registers remain zero, frame dumps are all-black, and
  the M32 300-frame audio digest is the SHA-256 of 959,128 zero bytes.
- Make scheduled CPU execution and core audio advancement share the same
  master-cycle timeline closely enough that YM2151 busy/status polling ages as
  ROM code executes. After this milestone, PacManV8 must progress past the FM
  init busy-poll loop and reach the documented T016/T017 review schedule.

Allowed scope:
- Core scheduler/emulation-loop changes inside `src/core/` that keep audio,
  VDP command advancement, CPU instruction execution, CPU T-state accounting,
  and event firing on one coherent master-cycle timeline.
- Narrow extracted Z180 opcode coverage inside `third_party/z180/` only for
  blockers exposed after the timing fix moves PacManV8 past the documented
  `PC 0x2B8B` YM2151 busy-poll loop. The first documented post-fix blocker is
  `ADD A,n` (`0xC6`) at `PC 0x2B1C`.
- Narrow audio-core changes inside `src/core/audio/` only if they are required
  to expose or preserve correct elapsed-master-cycle behavior for YM2151 busy
  state aging. Do not change chip synthesis, mixing math, resampling math, or
  PSG/MSM5205 behavior unless the timing fix proves a documented bug in elapsed
  time delivery.
- Tests that prove:
  - YM2151 busy-poll loops can make progress when CPU instructions execute
    inside a scheduler event slice.
  - `Bus::run_audio()` is not double-counted and audio output sample counts
    remain master-cycle-derived.
  - Existing deterministic tests remain stable or are intentionally updated
    where their prior expected value was proven to encode all-zero silence.
  - PacManV8 no longer remains at `PC 0x2B8B` with all-zero PCM and all-zero
    VDP display registers through the M32 review window.
- Documentation updates that replace the invalid M32 silence digest guard with
  the corrected post-M33 PacManV8 evidence.

Explicit non-goals:
- Frontend playback changes. M32 already added the SDL queue-path work; this
  milestone fixes the core timing blocker beneath it.
- New audio features, UI controls, WAV/export/recording support, spectrum
  displays, per-chip volume controls, or a second mixer/resampler path.
- VDP rendering feature work, input changes, debugger UI changes, cartridge
  mapper changes, or broad CPU opcode completion.
- Changing YM2151, AY-3-8910, or MSM5205 synthesis behavior to fake audible
  output. Audio must become nonzero because ROM-visible hardware timing lets
  the ROM reach its documented writes.
- Preserving the M32 PacManV8 300-frame hash
  `0d634fb059d15b12c4a8faf2412fbe08b85d187a31b1f22278ce3662f3b44390`; that
  value has been proven to hash all-zero PCM and is not a valid audible-output
  regression target.

Required tests:
- A focused scheduler/core test with a ROM-shaped YM2151 busy-poll sequence
  (`OUT (0x40),A`; repeated `IN A,(0x40)` / mask / `JR NZ`) proving that the
  busy bit clears after the documented elapsed master cycles while the CPU is
  executing inside one frame.
- A regression proving the same loop remains deterministic across repeat runs:
  same completed frame count, final PC/register state, event-log digest, audio
  sample count, and audio byte digest.
- A guard that audio sample counts still match the master-cycle timeline after
  the co-scheduling change, including a no-ROM or HALT-heavy case to catch
  double-counting.
- PacManV8 diagnostic regression for the M32 blocker:
  - At the chosen review frame count, CPU `PC` is no longer stuck at `0x2B8B`.
  - `AudioMixer::output_bytes()` contains at least one nonzero byte.
  - A frame dump contains at least one nonzero RGB byte, or a documented later
    frame count is selected if the ROM intentionally remains black during
    earlier initialization.
  - The new post-fix audio SHA-256 is recorded in the test/task summary and
    replaces the M32 all-zero digest guard.

Exit criteria:
- `cmake-build-debug/src/vanguard8_headless --rom
  /home/djglxxii/src/PacManV8/build/pacman.rom --frames 300 --hash-audio`
  no longer produces the all-zero PCM digest
  `0d634fb059d15b12c4a8faf2412fbe08b85d187a31b1f22278ce3662f3b44390`.
- PacManV8 progresses past the YM2151 busy-poll blocker at `PC 0x2B8B` and
  produces nonzero audio bytes during the review window.
- A headless frame dump at the selected review frame contains nonzero pixels,
  or the task documents an exact later frame where video first becomes
  nonblack and locks that value in tests.
- Existing `ctest` coverage passes with updated, intentional deterministic
  expectations.
- M32 can be unblocked afterward without further core scheduler changes.

Forbidden extras:
- Frontend, debugger UI, input, or unrelated VDP work.
- A fake audio unmute path, synthetic frontend-only sound, or any change that
  bypasses ROM-visible hardware timing.
- Keeping the old all-zero PacManV8 digest as an acceptance guard.
- Broad opcode completion, except for a narrowly documented opcode blocker that
  appears only after the timing fix lets PacManV8 progress farther.
