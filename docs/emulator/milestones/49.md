# Milestone 49 — Resolve HD64180 Internal I/O / Controller Port Collision

Status note, 2026-04-27 (active; M48 accepted):
- M48 was accepted on 2026-04-27. M49 was activated immediately
  afterward and addresses a separate, independently-discovered
  defect surfaced by the PacManV8 team during T025.
- The PacManV8 blocked task
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T025-per-frame-playing-tick.md`
  reports that the headless emulator's replay mechanism does not
  deliver controller state to the CPU's `IN A,(0x00)` instruction.
  The replay file specifies per-frame controller masks; the
  emulator's inspection report displays the correct masked state
  (e.g. `Controller 1 port: 0xEF` for RIGHT pressed); but
  `IN A,(0x00)` returns `0xFF` (all buttons released) regardless of
  the replay entry at that frame. Confirmed via the minimal repro
  documented in the blocker section: a 1-instruction test ROM that
  loops `IN A,(0x00)` and stores to `0x80F0`, peeked via
  `--peek-logical 0x80F0:1`, reads `0xFF` even when the active
  replay entry sets Controller 1 to `0xEF`.
- Root cause has been confirmed in the imported MAME HD64180 core.
  `third_party/z180/z180_core.cpp::Core::IN` (line 382-387) routes
  any port that satisfies `is_internal_io_address(port)` to
  `z180_readcontrol(port)` instead of the external
  `callbacks_.read_port` callback. `is_internal_io_address` (line
  321-325) returns true when `((port ^ m_iocr) & 0xffc0) == 0`.
  `m_iocr` is reset to `0x00`, so the internal I/O range covers
  `0x00-0x3F` at boot — which shadows the Vanguard 8's controller
  ports.
- The Vanguard 8 spec (`docs/spec/04-io.md:11-14`) places
  Controller 1 at port `0x00` and Controller 2 at port `0x01`,
  unconditionally. The HD64180 datasheet places its internal I/O
  range at the same addresses by default (ICR reset value covers
  `0x00-0x3F`). The two cannot coexist without an explicit
  hardware decision about how the Vanguard 8 keeps controller
  reads visible at those addresses.
- This collision was masked before M47 because the milestone-2-
  era hand-rolled timed dispatch always called `read_port` for
  `IN r,(n)` regardless of the internal-I/O comparator. The M47
  full-MAME-core import faithfully implements the HD64180
  internal-I/O carve-out, exposing a real spec gap that has been
  latent since milestone 0. M49 closes the gap.

Objective:
- Resolve the spec-level conflict between the controller port
  addresses (`0x00`, `0x01`) and the HD64180 default internal-
  I/O range (`0x00-0x3F`), then patch the emulator so that
  `IN A,(0x00)` and `IN A,(0x01)` deterministically deliver the
  bus-side controller state — including the replay-driven state
  applied via `Replayer::apply_frame` — for every CPU read along
  the timed dispatch path. Verify with the PacManV8 T025 minimal
  repro and unblock the PacManV8 T025 task.

Spec resolution (must land in the same change as the code):
- Decide the authoritative model for keeping controller reads
  visible at ports `0x00` / `0x01` despite the HD64180 default
  internal-I/O range. The implementer must pick exactly one of
  the following and document it in `docs/spec/01-cpu.md` and
  `docs/spec/04-io.md` before locking the code:
  1. **Reset-state ICR programming.** Document that the
     Vanguard 8 reset wiring programs the HD64180 ICR/IOCR so
     that the internal-I/O comparator window does not cover
     `0x00-0x3F`. Specify the exact reset value (e.g.
     `ICR = 0xC0`, internal-I/O window relocated to
     `0xC0-0xFF`), and pin it to the same authority style the
     spec already uses for `CBAR=0x48`, `CBR=0xF0`, `BBR=0x04`.
  2. **External-bus precedence at the controller ports.**
     Document that the Vanguard 8's external bus glue takes
     precedence at ports `0x00` and `0x01` over the HD64180
     internal-I/O response, and that no other port in the
     internal-I/O range is overlaid. (This is the option that
     does not require boot-ROM changes for any existing
     showcase ROM.)
  - The implementer should pick the option that is consistent
    with the rest of the spec, with real HD64180 documentation,
    and with the existing PacManV8/Vanguard8 showcase ROMs that
    already do `IN A,(0x00)` without an explicit ICR
    programming step. Spec drift is not an acceptable shortcut:
    the chosen model must be documented in the spec docs above
    *before* code lands.

Allowed scope:
- Spec edits in `docs/spec/01-cpu.md`, `docs/spec/04-io.md`, and
  `docs/spec/00-overview.md` to lock the chosen resolution.
- A narrow patch to the imported MAME core or its Vanguard 8
  adapter glue so that reads of ports `0x00` and `0x01` reach
  `bus_.read_port` and ultimately
  `ControllerPorts::read_port`. The patch must match the
  resolution chosen in the spec step:
  - For option 1, set `m_iocr` to the spec-pinned reset value at
    `Core::reset()` (or expose a Vanguard 8-side initialization
    hook), keep `is_internal_io_address` unchanged, and verify
    that `IN A,(0x00)` flows through `callbacks_.read_port`.
  - For option 2, add a narrow Vanguard 8-specific carve-out in
    the adapter (or in `Core::IN` / `Core::OUT` behind a hook)
    so that ports `0x00` and `0x01` always go to the external
    callback, and keep all other internal-I/O behavior intact.
- Symmetric handling for writes if the spec resolution requires
  it (e.g. so a future I/O-mapped output to those ports is not
  silently swallowed by `z180_writecontrol`); writes to `0x00`
  and `0x01` are currently undefined by the spec, so do not
  invent behavior — match whatever the spec resolution states.
- Replay-side verification only. No replay format change, no
  controller-port serialization change, and no
  `Replayer::apply_frame` redesign. The plumbing path
  (`headless.cpp:551-552` → `mutable_controller_ports()` →
  `ControllerPorts::port_state_` →
  `Bus::read_port(0x00)` →
  `ControllerPorts::read_port`) is already correct; only the
  CPU side needs to honor it.
- Tests and evidence:
  - A focused CPU-level test that constructs a Z180Adapter with
    a stub `read_port` and asserts that executing `IN A,(0x00)`
    invokes the callback with `port == 0x00` and that A receives
    the callback's return value. Repeat for `IN A,(0x01)`.
  - A bus-level integration test that drives the controller via
    `ControllerPorts::set_button(...)`, executes `IN A,(0x00)`
    on the timed core, and asserts A matches the active-low
    encoded port state.
  - A headless replay-driven repro reproducing the PacManV8
    T025 minimal repro: a tiny ROM that loops
    `IN A,(0x00) / LD (0x80F0),A`, paired with a one-frame
    replay file that sets Controller 1 to `0xEF`, peeked via
    `--peek-logical 0x80F0:1`. Expected post-fix: `0x80F0 = 0xEF`.
    Record the pre-fix and post-fix peek values as the
    M49-T01 evidence. Pin the test ROM and replay file under
    `tests/replays/` so the regression cannot silently come
    back.
  - A non-perturbation digest check: the replay-fixture frame-4
    SHA-256
    (`e46b5246bda293e09e199967b99ac352f931c04e2ad88e775b06a3b93ccb838c`,
    pinned in M47) and the M48-pinned T017 audio digest
    (`a765959a62e9a5afe9d075206efb8943e3141ef4274844a07c35f21c20d1ab27`)
    must remain byte-identical across three repeat runs. Neither
    fixture exercises the controller path, so the fix must not
    perturb them.
- Doc/task closure: append the M49 plan entry, append the M49
  task-index row, and produce a completion (or incompletion)
  summary in `M49-T01`.

Explicit non-goals:
- **No replay-format change.** The `.v8r` byte stream, frame
  layout, and the replay parser are out of scope. The defect is
  on the CPU read side; the replay side already loads the
  correct port state. Do not redesign the format.
- **No controller-state semantic change.** Active-low encoding,
  the 8-bit bit map (`docs/spec/04-io.md:18-27`), and the
  `Player::one` / `Player::two` mapping are unchanged.
- **No 4-player work.** The Vanguard 8 spec explicitly defers
  4-player expansion (`docs/spec/04-io.md:29-35`); M49 may not
  introduce ports for it, even speculatively.
- **No broadening of the HD64180 internal-I/O behavior.** Any
  internal-I/O register that the imported core already honors
  for ports outside `0x00`/`0x01` must continue to be honored.
  Do not relocate, disable, or reroute ASCI, CSI/O, timer, or
  DMA registers as a side effect.
- **No PacManV8 source edits from this repo.** PacManV8 is
  authoritative for its own ROM. The PacManV8 T025 task moves
  out of `blocked/` only after the Vanguard 8 fix lands and the
  PacManV8 team reruns their T025 evidence; that re-run is
  PacManV8 work, not M49 work.
- **No bundling of unrelated audit follow-ups.** Text 1 / Text 2
  renderers, Graphic 5 / 7, sprite overflow status, command-
  engine packing, and the Pac-Man palette VCLK closure stay in
  their own milestone slots.
- **No "fix the symptom only" workaround.** A patch that returns
  the controller value from `z180_readcontrol(0x00)` without
  resolving the spec-level collision is not acceptable — that
  silently invents behavior for the rest of the `0x00-0x3F`
  internal-I/O range.
