# Milestone 47 — Full MAME HD64180 Core Import (Retire Per-Opcode CPU Milestones)

Status note, 2026-04-26:
- Milestone 46 closed the in-scope ALU register/immediate timed
  opcode tail and was accepted on 2026-04-24. The PacManV8 T021
  harness passes end-to-end against M46. No new T021-driven CPU
  blocker is open.
- A repository audit (`docs/emulator/16-rom-readiness-audit.md`,
  2026-04-26) documented that the timed Z180 dispatch in
  `third_party/z180/z180_core.{hpp,cpp}` and
  `src/core/cpu/z180_adapter.cpp` is a milestone-2 boot stub that
  has been extended one opcode per milestone since M19. The audit
  enumerated concrete missing surfaces that no agentically-authored
  ROM and no standard Z80/HD64180 toolchain output can avoid:
  - All `IX` / `IY` (`DD` / `FD`) prefixes — including any frame-
    pointer-using subroutine prologue.
  - All block transfer / search instructions (`LDI` / `LDIR` /
    `LDD` / `LDDR` / `CPI` / `CPIR` / `CPD` / `CPDR`).
  - All eight `RST n` vectors (only `RST 0x38` reachable as IM1
    interrupt entry; the eight-byte `RST` opcodes are not
    dispatched).
  - All conditional `CALL cc,nn` (`0xC4`, `0xCC`, `0xD4`, `0xDC`,
    `0xE4`, `0xEC`, `0xF4`, `0xFC`).
  - Conditional `JP cc,nn` for `C`, `NC`, `PO`, `PE`, `P`, `M`
    (only `JP Z,nn` and `JP NZ,nn` are wired).
  - `JP (HL)` (`0xE9`), the standard jump-table dispatch primitive.
  - Indirect loads `LD A,(BC)` (`0x0A`), `LD A,(DE)` (`0x1A`),
    `LD (BC),A` (`0x02`), `LD (DE),A` (`0x12`).
  - `EX (SP),HL` (`0xE3`), `EX AF,AF'` (`0x08`), `EXX` (`0xD9`),
    `LD SP,HL` (`0xF9`).
  - Accumulator rotates `RLCA` (`0x07`), `RLA` (`0x17`),
    `RRA` (`0x1F`).
  - `DAA` (`0x27`), `CCF` (`0x3F`), `LD (HL),n` (`0x36`).
  - 249 of 256 CB-prefix sub-opcodes (only seven CB ops dispatched
    today).
  - The ED-prefix block I/O, `NEG`, `RETN`, `IM 0` / `IM 2`,
    `IN r,(C)` / `OUT (C),r`, `RLD` / `RRD`, `INI` / `OTIR` and
    siblings.
- The user has decided on 2026-04-26 to retire the one-opcode-per-
  milestone pattern entirely. The Vanguard 8 *is* an HD64180
  system, so the natural posture is to import the **full** MAME
  z180 core in one pass and shrink `Z180Adapter` to bus glue. The
  user has explicitly rejected:
  - Continuing to add opcodes one milestone at a time.
  - Hand-rolling a Z80 core plus a separate HD64180 extension
    layer (option 2 from the audit's "Recommended order of work"
    discussion). All HD64180 functionality must be available by
    default.
- This milestone supersedes the milestone-2 extraction scope
  contract recorded in `third_party/z180/README.md`. After M47
  lands, "full opcode coverage" is no longer a non-goal; it is the
  shipped state.

Objective:
- Replace the hand-written Vanguard 8 Z180 stub with the full MAME
  HD64180 / Z180 CPU core (BSD-3-Clause, pinned MAME upstream
  commit), removing the per-opcode timed dispatch tables in
  `Z180Adapter` and any throw site of the form
  `"Unsupported timed Z180 opcode …"`. After M47, every published
  HD64180 instruction (full Z80 base set plus `MLT`, `IN0`,
  `OUT0`, `TST`, `TSTIO`, `OTIM` / `OTDM` / `OTIMR` / `OTDMR`,
  `SLP`) executes without dispatch fallthrough, with cycle timing
  supplied by the imported core, not by hand-rolled adapter
  tables.

Allowed scope:
- Replace `third_party/z180/z180_core.hpp` and
  `third_party/z180/z180_core.cpp` with a faithful import of the
  MAME HD64180 / Z180 CPU device sources (`src/devices/cpu/z180/`
  in the upstream tree) at a single pinned commit. The imported
  core must include:
  - The full Z80 instruction set, including `DD` / `FD` (`IX` /
    `IY`), `CB`, and `ED` prefixes.
  - The HD64180-specific instructions (`MLT`, `IN0`, `OUT0`,
    `TST`, `TSTIO`, `OTIM`, `OTDM`, `OTIMR`, `OTDMR`, `SLP`).
  - The on-chip MMU registers (`CBAR`, `CBR`, `BBR`) with the
    Vanguard 8 spec's logical-to-physical translation rules
    preserved.
  - The HD64180 PRT timer block (`TMDR0/1`, `RLDR0/1`, `TCR`)
    with the spec-documented vectored interrupt semantics.
  - The HD64180 DMA controller surface used by the existing
    `Z180Adapter::execute_dma()` callers, or an equivalent that
    keeps the externally-visible behavior of the bus and tests
    unchanged.
  - The vectored-interrupt response for `INT1` / `INT2` /
    `PRT0` / `PRT1` driven by `I` and `IL`, matching the spec's
    `vector_low = (IL & 0xE0) | fixed_code` rule.
- The MAME device-framework glue (`device_t`, `device_execute_
  interface`, `state_string_export`, save-state nodes, etc.)
  must be stripped or replaced with a small in-tree shim that
  preserves the existing `vanguard8::core::cpu::Z180Adapter`
  public surface used by `Bus`, the scheduler, the debugger, and
  the save-state and replay subsystems.
- Update `src/core/cpu/z180_adapter.{hpp,cpp}` to delegate to the
  imported core for every instruction. Specifically:
  - Delete `Z180Adapter::current_instruction_tstates()`,
    `Z180Adapter::cb_instruction_tstates()`, and
    `Z180Adapter::ed_instruction_tstates()` — including every
    range-check, every `case 0x..` opcode label, and every
    `throw std::runtime_error("Unsupported timed Z180 opcode …")`
    site. The imported core supplies its own cycle counts.
  - Keep the existing observability surface: breakpoints,
    `record_breakpoint_hit`, `record_bank_switch` on `OUT0
    (0x39)`, `bank_switch_log`, `acknowledge_int1`, internal
    I/O mirroring, save-state snapshot/load, peek/translate, and
    HALT-resume behavior. Where the imported core represents
    state differently, write a translation layer in the adapter,
    not a behavior change.
  - Preserve `CpuStateSnapshot` field semantics so existing save
    states and replay fixtures continue to deserialize. If the
    imported core's internal layout differs, version the snapshot
    structure and include a one-time migration path.
- Update `tests/test_cpu.cpp` to:
  - Delete every test that asserts a per-opcode T-state value
    against the hand-rolled `current_instruction_tstates()`
    family (these tests were proxies for the old dispatch tables
    and will be made obsolete by the import).
  - Replace them with a small set of behavioral / spec-anchored
    tests: MMU translation through `CBAR` / `CBR` / `BBR`,
    illegal `BBR` rejection, INT0 IM1 dispatch, INT1 vectored
    dispatch via `I` / `IL`, PRT0 / PRT1 vectored dispatch, the
    HD64180 MMU bank-switch event recording on `OUT0 (0x39)`, and
    HALT-resume on interrupt. These are the spec-pinned
    invariants — leave the imported core to be authoritative for
    individual opcode semantics.
  - Add coverage proving that previously unsupported instructions
    now execute. At a minimum: an `LDIR` block-copy program, an
    `IX`-based subroutine prologue (`PUSH IX` / `LD IX,nn` /
    `LD A,(IX+d)`), and a CB-prefix bit op (`BIT 0,B` /
    `RES 4,(HL)` / `SET 7,A`). These are diagnostic, not
    exhaustive — the imported core's correctness is taken from
    its upstream provenance, not re-proven here.
- Re-pin the deterministic replay fixture digests in
  `tests/replays/`. More accurate timing from the imported core
  is expected to shift frame and event-log hashes by a one-time
  delta. The new digests must be recorded as the post-M47
  baseline; the shift itself is in scope, but the *direction* of
  any digest change must be inspected against the recorded fixture
  ROM and noted in the task file.
- Update `third_party/z180/README.md` to state the new scope:
  full MAME HD64180 / Z180 import, pinned commit, license, and
  a note that "full opcode coverage" is the shipped state, not a
  non-goal. Remove the milestone-2 opcode list.
- Update `docs/emulator/07-implementation-plan.md` to:
  - Append M47 with this contract.
  - Mark the per-opcode milestones M19 through M46 as superseded
    in spirit by M47, but do not rewrite their history. The
    closure annotation lives in M47's plan entry, not in the
    older milestone bodies.
- Update `docs/emulator/16-rom-readiness-audit.md` Section 1's
  status entry once M47 closes (the "Blocker" severity is
  expected to drop to "Resolved").

Explicit non-goals:
- **No VDP, audio, controller, scheduler, debugger, save-state
  format, headless format, or frontend changes**, except the
  minimal save-state snapshot versioning needed to migrate the
  CPU register block. If a non-CPU subsystem requires changes to
  accommodate the imported core, that is a defect in this
  milestone's adapter shim, not a license to broaden the scope.
- **No PacManV8 source edits from this repo.** The PacManV8 T021
  harness is a verification consumer; if its expectations need to
  shift after the M47 import, that is PacManV8's milestone.
- **No re-implementation of the MAME core's opcodes.** Adapter
  glue calls into the imported core; it does not re-derive
  semantics. Bug fixes against MAME's own behavior are out of
  scope and would be an upstream submission, not a Vanguard 8
  patch.
- **No partial Z80 + hand-rolled HD64180 extras combination.**
  This milestone explicitly forbids landing a Z80-only third-
  party core and bolting `MLT` / `IN0` / `OUT0` / `TST` /
  `TSTIO` / `OTIM*` / `SLP` onto it locally. The single
  authorized path is the full MAME HD64180 / Z180 import.
- **No attempt to keep the milestone-2 extraction stub as a
  fallback.** After M47 lands, the old `z180_core.hpp` /
  `z180_core.cpp` content does not exist anywhere in the tree
  (no commented-out vestige, no `#ifdef LEGACY_Z180_STUB`). The
  imported core is the only core.
- **No undocumented HD64180 behavior invention.** Where MAME's
  upstream behavior diverges from the Vanguard 8 spec
  (`docs/spec/01-cpu.md`), the spec wins and the divergence is
  documented in the adapter, exactly as the milestone-2
  extraction note already required for `CBR` / `BBR` / `CBAR`
  reset values and the `IL & 0xE0` vector-pointer rule.

Required tests:
- Spec-anchored CPU integration tests (kept or rewritten to suit
  the imported core, but pinned to the spec, not to the hand-
  rolled dispatch tables):
  - MMU translation correctness across the boot configuration
    (`CBAR=0x48`, `CBR=0xF0`, `BBR=0x04`) and at least one
    runtime bank switch.
  - Illegal `BBR ≥ 0xF0` rejection / warning behavior.
  - `OUT0 (0x39),A` recorded in `bank_switch_log`.
  - INT0 IM1 dispatch reaches `0x0038` and the configured V-blank
    handler clears `S#0` bit 7.
  - INT1 vectored dispatch via `I = 0x80`, `IL = 0xE0` reads the
    handler word at `0x80E0` / `0x80E1`.
  - PRT0 / PRT1 vectored dispatch via the same `I` / `IL` table.
  - HALT-resume after an interrupt (M36 invariant).
- New coverage proving the previously-unsupported instruction
  classes now execute end-to-end:
  - `LDIR` block copy of N bytes from one logical address to
    another, with `BC = 0` post-condition and the destination
    region byte-equal to the source.
  - An `IX`-based subroutine that uses `PUSH IX` / `LD IX,SP+0`-
    style frame setup and `LD A,(IX+d)` to read a stack local.
  - A CB-prefix bit op set/test cycle (`SET n,r`, `BIT n,r`,
    `RES n,r`) that exercises the `H` flag side effects.
  - A `JP (HL)` jump-table dispatch with at least three entries.
  - A `RST 0x08` round-trip (push, jump to 0x0008, return).
  - `IM 2` followed by an external-INT2 vector fetch (uses the
    HD64180 mode-independent vectored response, but accepts the
    `IM 2` opcode itself without trapping).
- Negative guard: confirm that no opcode causes the runtime to
  throw `"Unsupported timed Z180 opcode …"` from
  `Z180Adapter::*_instruction_tstates`. Implementation-wise this
  is satisfied by deleting those throw sites; the test should
  assert their absence, e.g. by walking every byte 0x00–0xFF as a
  primary opcode through the adapter's timing API and asserting
  no throw is observed.
- Replay non-perturbation guard: the existing fixture replay must
  reproduce its (newly re-pinned) digests on three repeat runs
  with byte-identical results.
- Canonical PacManV8 T021 harness:
  `python3 tools/pattern_replay_tests.py
   --evidence-dir tests/evidence/T021-pattern-replay-and-fidelity-testing`
  must run end-to-end without an `Unsupported timed Z180 opcode`
  trap and either pass its assertions or surface a non-CPU
  blocker (recorded in the task file). M47 does not depend on
  PacManV8 fidelity assertions passing — only on the absence of
  CPU-dispatch traps.

Exit criteria:
- `third_party/z180/z180_core.hpp` and `z180_core.cpp` are the
  imported MAME HD64180 / Z180 core (or its in-tree shim) with
  the upstream commit and license recorded in
  `third_party/z180/README.md`.
- `Z180Adapter::current_instruction_tstates()`,
  `Z180Adapter::cb_instruction_tstates()`,
  `Z180Adapter::ed_instruction_tstates()`, and every
  `"Unsupported timed Z180 opcode"` throw site no longer exist
  anywhere in `src/core/cpu/`.
- `cmake --build cmake-build-debug` succeeds.
- `ctest --test-dir cmake-build-debug --output-on-failure`
  passes with the new test surface in place. Replay-digest
  re-pinning is recorded in the task file with the pre/post hash
  pair and a one-line rationale ("expected: imported core
  applies more accurate timing than the hand-rolled tables").
- `cmake-build-debug/src/vanguard8_headless --rom <fixture>` runs
  the replay fixture to its programmed end without dispatch
  traps.
- The PacManV8 T021 harness either passes end-to-end or surfaces
  a non-CPU blocker recorded in the M47-T01 task file.
- `docs/emulator/16-rom-readiness-audit.md` Section 1 is updated:
  severity drops to "Resolved" with a one-line note pointing at
  M47.

Forbidden extras:
- Reintroducing per-opcode milestones for the HD64180 surface.
  After M47, any future ROM trap on a CPU dispatch path is a bug
  in the adapter or in the imported core's pin, **not** an
  invitation to open M48 for that single opcode.
- Vendoring a Z80-only third-party core and writing a local
  HD64180 extension layer.
- Changing the spec to match upstream MAME behavior. The spec
  remains authoritative; adapter shim notes are where any
  divergence is recorded.
- Bundling unrelated audit closures (Text 1 / Text 2 renderers,
  Graphic 5 / 7, sprite overflow status fields, command-engine
  packing for non-Graphic-4 modes). Those are separate
  milestones; M47 does not bundle them in even though they are
  also enumerated in the ROM-readiness audit.
- Editing PacManV8 sources, harness, or evidence to route around
  any post-import behavior change.
