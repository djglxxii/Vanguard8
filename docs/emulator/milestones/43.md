# Milestone 43 — Logical Peek Row Prefix Format for PacManV8 T021

Objective:
- Close the next real-ROM T021 blocker exposed after M42 added the
  authorized `SCF` (`0x37`) timed opcode coverage. With SCF running,
  the PacManV8 T021 `pattern_replay_tests.py` harness no longer
  aborts at `PC=0x1315`; the headless binary now runs to the
  requested `--inspect-frame` checkpoint and produces a valid
  inspection report. The harness then fails with:

  ```text
  pattern_replay_tests.py error: inspection report did not contain logical 0x8270:13
  ```

- The root cause is a format mismatch in Vanguard 8's own inspection
  output, not a Z180 opcode gap and not a PacManV8 authoring
  issue. `src/frontend/headless_inspect.cpp::append_byte_row` is
  shared by both the `[peek-mem]` (physical, 20-bit) and
  `[peek-logical]` (logical, 16-bit) paths, and formats every row
  display address with `hex20` (five hex digits), so a logical peek
  at `0x8270` emits rows like:

  ```text
  [peek-logical]
  logical 0x8270 physical 0xf0270 region ca1 length 13
    0x08270: 01 fe 24 00 00 00 00 00 0e 1a 00 00 04
  ```

  The PacManV8 T021 harness regex is
  `BYTE_ROW_PATTERN = r"^\s+0x([0-9a-f]{4}):((?: [0-9a-f]{2})+)$"`
  — it expects exactly four hex digits after the `0x`, matching
  the 16-bit logical address. A logical row prefixed with
  `0x08270` (five digits) silently fails to match, the harness
  collects zero bytes, and `parse_peek_bytes` raises
  `inspection report did not contain logical 0x{address:04x}:{length}`.
  This is a latent bug that was masked by every earlier T021 run
  aborting on an unsupported-opcode exception before the
  inspection parser got a chance to run.

- Logical addresses are 16-bit by design (the header line
  correctly prints `logical 0x8270`, `region ca1`, `length 13`).
  The row-prefix width should match the header's logical width
  (4 hex digits) on the `[peek-logical]` path, and must remain
  5 hex digits on the `[peek-mem]` path, where the display
  address is a 20-bit physical address. The header-level
  `physical 0xfxxxx` cross-reference continues to let the
  harness (or a human) correlate the two.

Explicit non-goals:

  handlers, or adapter T-state tables.
- **No changes to the inspection report header line format.**
  The header line `logical 0x8270 physical 0xf0270 region ca1
  length 13` is already well-formed and is what the harness uses
  to anchor the block; only the subsequent byte-row prefix
  changes.
- **No changes to the physical peek path or `[peek-mem]`
  output.** Physical rows must still print 5 hex digits.
- **No changes to VRAM dump formatting, event log formatting,
  VDP register dumps, or any other observability surface.**
- **No PacManV8 harness edits** to widen its regex. The
  authoritative fix is emitter-side, because the logical
  peek's display address is genuinely 16-bit.
- **No interrupt-controller, scheduler, audio, VDP, debugger,
  save-state, or core emulator changes.** The core emulator is
