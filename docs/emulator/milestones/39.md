# Milestone 39 — Timed HD64180 16-bit Pair INC/DEC and CB-Prefix Coverage for PacManV8 T021

Objective:
- Close the next real-ROM timed CPU compatibility gap exposed after M38
  unblocked the carry-conditional `RET cc` path. With the M37/M38 timed
  load and conditional-return surface now landed, the PacManV8 blocked
  task
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T021-pattern-replay-and-fidelity-testing.md`
  reaches its replay-validation entry path and the canonical headless
  runtime aborts with:

  ```text
  terminate called after throwing an instance of 'std::runtime_error'
    what():  Unsupported timed Z180 opcode 0xB at PC 0x1209
  ```

- `PC=0x1209` is inside `collision_init`. The failing instruction is
  standard Z80 `DEC BC` (`BC <- BC - 1`, `PC += 1`, 6 T-states, flags
  untouched). T021 genuinely needs it as the per-cell countdown driving
  the pellet-bitset clear loop. Rewriting `collision_init` to avoid the
  16-bit decrement would be an out-of-scope emulator workaround.

- The blocker's look-ahead for related active-path opcodes confirms two
  more sister 16-bit pair INC/DEC forms are reached as soon as the
  `DEC BC` gap is closed:
  - `INC DE` (`0x13`) — used in `collision_init` to advance to the next
    pellet-bitset byte after each 8-cell mask wrap.
  - `DEC HL` (`0x2B`) — used by `ghost_mode_tick` /
    `ghost_mode_tick_frightened` and the score / dot bookkeeping paths
    reached by replay validation.

- The same look-ahead confirms the CB-prefix dispatch is entirely
  missing from the timed core (no `0xCB` entry in
  `third_party/z180/z180_core.cpp` and no `0xCB` arm in
  `src/core/cpu/z180_adapter.cpp`). The active T021 replay path is
  expected to fall straight into a CB-prefix abort once the 16-bit
  INC/DEC gap is closed, against these specific sub-opcodes:
  - `SRL A` (`CB 3F`) — used by existing movement / collision code
    already exercised by replay validation.
  - `BIT 4,A` (`CB 67`), `BIT 5,A` (`CB 6F`), `BIT 6,A` (`CB 77`),
    `BIT 7,A` (`CB 7F`) — used by the new replay input decoder added
    by PacManV8 T021 to unpack joypad-bit fields from each input byte.

  `RES`, `SET`, the rotate/shift forms other than `SRL`, the `(HL)`
  memory-target forms, the `BIT b,r` forms for registers other than
  `A`, and the index-register CB variants (`DD CB`, `FD CB`) are **not**
  on any confirmed T021 active path and are explicitly out of scope.

- This milestone exists to close the confirmed `DEC BC` gap and — to
  avoid another rebuild-per-opcode loop on the same active path — the
  documented look-ahead 16-bit pair INC/DEC form `INC DE` and `DEC HL`,
  the closest BC-pair sister `INC BC` (closing the BC half of the
  family the way M38 closed the carry half of `RET cc`), and the
  narrow CB-prefix dispatch surface needed by the listed sub-opcodes.
  This matches the family-closure precedent established by milestones
  34, 37, and 38: close the immediately-implicated forms plus the
  closest sister forms in the same family, and stop there.

Allowed scope:
- Timed opcode coverage inside the existing extracted HD64180 runtime
  for the missing 16-bit register-pair INC/DEC forms directly
  implicated by the T021 blocker and its documented active-path
  look-ahead:
  - `DEC BC` (`0x0B`) — the confirmed failing opcode at `PC 0x1209`.
  - `INC DE` (`0x13`) — look-ahead, `collision_init` pellet-byte
    advance.
  - `DEC HL` (`0x2B`) — look-ahead, ghost-mode and bookkeeping paths.
  - `INC BC` (`0x03`) — sister form of `DEC BC`, closed in this pass
    to avoid another single-opcode rebuild cycle on the BC half of the
    16-bit pair INC/DEC family.
- Implementation shape for the 16-bit pair INC/DEC opcodes must follow
  the existing pattern already used for `op_dec_de` (`0x1B`) and
  `op_inc_hl` (`0x23`) in `third_party/z180/z180_core.cpp`: a single
  16-bit add or subtract on the target pair, no flag updates, `PC += 1`.
- Adapter T-state timing entries for each new 16-bit pair INC/DEC
  opcode added in this milestone: **6 T-states**, registered alongside
  the already-covered `0x1B` and `0x23` group in
  `src/core/cpu/z180_adapter.cpp`.
- Narrow CB-prefix dispatch surface inside the existing extracted
  HD64180 runtime, restricted to the sub-opcodes confirmed by the T021
  blocker:
  - `SRL A` (`CB 3F`) — shift-right-logical on the accumulator: bit
    0 → carry, bit 7 ← 0, then update S/Z/P flags from the result;
    H and N cleared.
  - `BIT 4,A` (`CB 67`), `BIT 5,A` (`CB 6F`), `BIT 6,A` (`CB 77`),
    `BIT 7,A` (`CB 7F`) — Z flag set from the inverse of the tested
    bit, H set, N cleared, S and PV per the standard `BIT b,r`
    semantics for `r=A`, C unchanged, A unchanged.
  - The CB-prefix dispatch handler must register against `opcodes_[0xCB]`
    in `Core::install_opcode_table()`, fetch the next byte as the
    sub-opcode, and route only the listed CB sub-opcodes. Any other CB
    sub-opcode encountered must throw the existing
    `Unsupported timed Z180 opcode 0xCB <subop> at PC ...` style of
    `std::runtime_error`, preserving the current "fail fast on
    untimed" contract used by the rest of the adapter.
- Adapter T-state timing entry for the CB prefix dispatch: route on
  `case 0xCB:` to a new helper `cb_instruction_tstates(subop)` that
  returns **8 T-states** for `SRL r` register forms and **8 T-states**
  for `BIT b,r` register forms (matching the Z80 timing convention
  already in use by the adapter; see the existing 6-T-state entries
  for `0x1B` / `0x23`, which mirror Z80 rather than Z180 published
  cycles). Sub-opcodes outside the allowed list must raise the same
  unsupported-opcode runtime error pattern used by the main dispatch.
- Tests and docs needed to lock the newly covered timed opcode surface.
- Narrow runtime/integration coverage proving the canonical PacManV8
  T021 minimal repro can run past `PC=0x1209` without a timed-opcode
  abort, and that the replay validation harness reaches its first
  authored checkpoint.

Explicit non-goals:
- **No general CPU opcode sweep.** No new arithmetic families,
  index-register (`DD`/`FD`) prefixes, broader `CB`-prefix work, or
  block opcodes.
- **No other 16-bit pair INC/DEC forms.** `INC SP` (`0x33`) and
  `DEC SP` (`0x3B`) are not on any confirmed active path in the T021
  blocker or its look-ahead and are explicitly out of scope. They will
  be addressed in a separate milestone if and when a real-ROM path
  exercises them.
- **No broader CB-prefix coverage.** `RLC`, `RRC`, `RL`, `RR`, `SLA`,
  `SRA`, `SLL` shift/rotate forms; `RES b,r`; `SET b,r`; `BIT b,r` for
  any `r` other than `A`; `BIT b,(HL)`, `RES b,(HL)`, `SET b,(HL)`,
  and any `(HL)`-target shift/rotate; and the index-register CB
  variants (`DD CB d <op>`, `FD CB d <op>`) are all out of scope.
- **No rewrite of existing 16-bit pair INC/DEC handlers.** `0x1B` and
  `0x23` are already timed and must not be touched.
- **No interrupt-controller, scheduler, audio, VDP, or frontend
  changes.** This milestone is exclusively about adding the missing
  16-bit pair INC/DEC opcodes and the narrow CB-prefix dispatch
  surface enumerated above.
- **No PacManV8 ROM edits.** The T021 evidence refresh happens in the
  PacManV8 repo after this milestone lands.
- **No M35 observability-flag changes.** `--inspect`, `--inspect-frame`,
  `--peek-mem`, `--peek-logical`, `--dump-cpu`, `--dump-vdp-regs`,
  `--dump-vram-a`, `--dump-vram-b`, and `--run-until-pc` must behave
  identically before and after this milestone.
- **No HALT-wake changes.** M36 owns that path; its tests must stay
  green unchanged.
- **No changes to `LD r,n` coverage from M37 or to `RET cc` coverage
  from M38.** Those handlers and their adapter T-state entries must
  remain untouched.

Required tests:
- Focused timed CPU coverage in `tests/test_cpu.cpp` for each new
  16-bit pair INC/DEC opcode, pinning:
  - `DEC BC` (`0x0B`): after execution, `BC == BC_before - 1` (with
    16-bit wrap from `0x0000` to `0xFFFF`), all other registers
    unchanged, `PC` advanced by 1, all flags (S/Z/H/PV/N/C) untouched,
    adapter reports 6 T-states.
  - `INC BC` (`0x03`): same contract, `BC == BC_before + 1` (with
    16-bit wrap from `0xFFFF` to `0x0000`).
  - `INC DE` (`0x13`): same contract for the DE pair, including the
    `0xFFFF → 0x0000` wrap edge.
  - `DEC HL` (`0x2B`): same contract for the HL pair, including the
    `0x0000 → 0xFFFF` wrap edge.
- At least one test mirroring the PacManV8 `collision_init` pattern
  around `PC=0x1209`: a sequence that loads a non-trivial value into
  `BC`, executes `DEC BC`, then branches on the resulting register
  state via an existing already-timed opcode (`OR A` / `LD A,B` / etc.)
  to confirm the decrement participates correctly in the surrounding
  loop.
- Focused timed CPU coverage in `tests/test_cpu.cpp` for each allowed
  CB-prefix sub-opcode, pinning:
  - `SRL A` (`CB 3F`): `A` shifted right by one, bit 7 cleared, old
    bit 0 → C, S cleared (always, since bit 7 ends at 0), Z reflects
    new `A == 0`, H cleared, PV reflects parity of new `A`, N
    cleared; `PC` advanced by 2; adapter reports 8 T-states.
  - `BIT 4,A` (`CB 67`): Z set to inverse of bit 4 of `A`, H set, N
    cleared, S and PV per the standard `BIT b,r` semantics for `r=A`,
    C unchanged, `A` unchanged; `PC` advanced by 2; adapter reports 8
    T-states. The same contract is exercised for `BIT 5,A` (`CB 6F`),
    `BIT 6,A` (`CB 77`), and `BIT 7,A` (`CB 7F`) with the
    corresponding bit index.
- A negative test confirming that an out-of-scope CB sub-opcode
  (pick one: `RES 0,B` (`CB 80`)) raises the existing
  `Unsupported timed Z180 opcode` runtime error pattern with the CB
  sub-opcode reported in the message, so the "fail fast on untimed"
  contract is preserved.
- A negative test confirming that an out-of-scope main-table opcode
  in the same numerical neighborhood (pick one: `INC SP` (`0x33`))
  still raises the existing unsupported-opcode runtime error and is
  not silently routed to a 16-bit pair INC/DEC handler.
- Adapter-level T-state coverage confirming each new opcode is
  classified in the correct timing case (6 T-states for the 16-bit
  pair INC/DEC forms; 8 T-states for the CB-prefix sub-opcodes via
  `cb_instruction_tstates`) and that the overall adapter dispatch
  remains deterministic.
- Non-perturbation guard: for a chosen ROM, `--frames`, and hashing
  selection that does not exercise the new opcodes, frame and audio
  digests before and after this milestone must be byte-identical.
  Reuse an existing M33 / M35 / M37 / M38-verified fixture already
  present under `tests/`.
- Runtime/integration coverage that rebuilds the PacManV8 ROM and
  proves the canonical T021 minimal repro

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 T021 path
  with `Unsupported timed Z180 opcode 0xB at PC 0x1209` (or at any
  `INC DE`, `DEC HL`, `INC BC`, `SRL A`, or `BIT 4..7,A` site reached
  by this ROM along the replay validation path).
- The added timed opcode coverage for `DEC BC`, `INC BC`, `INC DE`,
  `DEC HL`, the CB-prefix dispatch handler, and the listed CB
  sub-opcodes (`SRL A`, `BIT 4..7,A`) is narrow, documented, and
  regression-covered, with explicit negative tests for at least one
  out-of-scope CB sub-opcode and one out-of-scope 16-bit pair INC/DEC
  opcode.
- `ctest --test-dir cmake-build-debug --output-on-failure` passes with
  the new opcode tests included, and no pre-existing CPU test is
  relaxed or removed.
- Deterministic digests for ROMs that do not depend on the new opcodes
  are unchanged.
- PacManV8 T021 becomes actionable again from the emulator side; the
  actual replay-checkpoint evidence (pattern vectors, frame hashes,
  PPM dumps) and acceptance still happen in the PacManV8 repo.

Forbidden extras:
- Turning this milestone into a broad 16-bit pair INC/DEC sweep that
  pulls in `INC SP` (`0x33`) or `DEC SP` (`0x3B`) without a confirmed
  active-path use.
- Turning this milestone into a broad CB-prefix sweep, including any
  rotate/shift form other than `SRL A`, any `BIT b,r` form for `r ≠ A`,
  any `BIT b,(HL)` form, any `RES`/`SET` form, or any
  `DD CB` / `FD CB` indexed CB variant.
- Bundling unrelated PacManV8 compatibility fixes under the same task.
- Loosening existing timed CPU or integration tests instead of adding
  the missing opcode coverage alongside them.
- Altering PacManV8 sources to route around the blocker.
- Adding silent fallthroughs in the new CB dispatch instead of
  preserving the existing fail-fast `std::runtime_error` contract for
  unsupported sub-opcodes.
