  was moved to blocked when the canonical PacManV8 T021 replay
  harness advanced past the AND sites and reached a new

    what():  Unsupported timed Z180 opcode 0xD6 at PC 0x3EA

  `0xD6` is `SUB n`, used by PacManV8
  `movement_distance_to_next_center_px` on the

  `sub MOVEMENT_TILE_CENTER`). `SUB n` is also used at

Objective:
- Close the full ALU register/immediate timed-opcode tail in a
  single Vanguard8 patch, matching the existing range-check style
  already used for `LD r,r'` (`0x40..0x7F`), `ADD A,r`
  (`0x80..0x87`), and `AND r` (`0xA0..0xA7`), so the PacManV8 T021
  replay path cannot abort again on a sister register-form or
  immediate-form ALU opcode inside the same rebuild cycle.

Allowed scope:
- Timed extracted-core coverage for the four register-form ALU
  families listed below, expressed as range-checks alongside the
  existing `LD r,r'`, `ADD A,r`, and `AND r` range-checks in
  `Z180Adapter::current_instruction_tstates()`:
  - `ADC A,r` (`0x88..0x8F`; `0x8E` = `ADC A,(HL)`)
  - `SUB r`   (`0x90..0x97`; `0x96` = `SUB (HL)`)
  - `SBC A,r` (`0x98..0x9F`; `0x9E` = `SBC A,(HL)`)
  - `XOR r`   (`0xA8..0xAF`; `0xAE` = `XOR (HL)`)
- Removal of the now-redundant explicit dispatch entries for
  `0x91` (`SUB C`), `0x93` (`SUB E`), and `0xAF` (`XOR A`) in the
  existing `return 4` switch block, so the new range-checks are the
  single source of truth for those opcodes. Do **not** touch the
  existing explicit `OR r` (`0xB0..0xB7`) or `CP r` (`0xB8..0xBF`)
  entries ŌĆö those ranges are already fully covered and are out of
  scope for this patch.
- Addition of the four missing immediate-arithmetic peers to the
  existing uniform-7-cycle `return 7` case-label group alongside
  `ADD A,n` (`0xC6`), `AND n` (`0xE6`), `OR n` (`0xF6`), and
  `CP n` (`0xFE`):
  - `ADC A,n` (`0xCE`)
  - `SUB n`   (`0xD6`) ŌĆö currently confirmed T021 blocker
  - `SBC A,n` (`0xDE`)
  - `XOR n`   (`0xEE`)
- Focused CPU coverage for every newly covered opcode (operand/
  result, flag behavior per published HD64180/Z80 semantics, `PC`
  advance, and adapter T-state classification; the `(HL)` forms
  additionally pin the memory-read path).
- Narrow non-perturbation fixture regression proof that existing
  fixture ROMs which do not exercise these opcodes remain byte-
  identical in frame/audio/event-log digests.
- Doc/task updates required to lock the new behavior.

Explicit non-goals:
- **No general CPU opcode sweep.** Only the four register-form ALU
  families above (`ADC A,r`, `SUB r`, `SBC A,r`, `XOR r`) and the
  four matching immediate peers (`ADC A,n`, `SUB n`, `SBC A,n`,
  `XOR n`) are in scope. Do not add rotate/shift families,
  index-register (`IX`/`IY`) forms, or ED-prefix forms unless a new
  T021 repro proves they are required ŌĆö and in that case, open the
  next milestone, do not broaden this one.
- **Do not touch `OR r` (`0xB0..0xB7`) or `CP r` (`0xB8..0xBF`).**
  Those ranges are already fully covered by the existing explicit
  case entries. The recommended fix explicitly excludes them from
  this patch.
- **No PacManV8 source edits from this repo.** PacManV8 ROM,
  harness, and evidence changes stay in the PacManV8 repo.
- **No unrelated subsystem work.** No VDP, audio, scheduler,
  debugger, save-state, desktop GUI, or broad frontend changes.
  No headless replay/inspection/reporting format changes unless a
  new T021 repro proves one is strictly required, in which case
  defer to a follow-up milestone.
- **No loosening of existing tests.** Do not relax the fail-fast
  `Unsupported timed Z180 opcode` contract for any opcode outside
  this milestone's allowed scope.
- **No undocumented behavior invention.** The `ADC`, `SUB`, `SBC`,
  and `XOR` families follow published HD64180/Z80 semantics. Cycle
  counts follow the same Z180 ALU timing already used by the
  existing `ADD A,r` and `AND r` range-checks (4 T-states for
  register forms, 7 T-states for `(HL)` and immediate forms). No
  Z180-specific deviations apply to these families.

Required tests:
- Focused timed CPU coverage in the existing CPU test file
  (`tests/test_cpu.cpp`), following the pattern already used for
  the `SUB C` / `SUB E` and `AND r` tests, asserting that
  `current_instruction_tstates()` returns:
  - **4** for each register form of:
    - `ADC A,r` (`0x88..0x8D`, `0x8F`)
    - `SUB r`   (`0x90..0x95`, `0x97`)
    - `SBC A,r` (`0x98..0x9D`, `0x9F`)
    - `XOR r`   (`0xA8..0xAD`, `0xAF`)
  - **7** for each `(HL)` form:
    - `ADC A,(HL)` (`0x8E`)
    - `SUB (HL)`   (`0x96`)
    - `SBC A,(HL)` (`0x9E`)
    - `XOR (HL)`   (`0xAE`)
  - **7** for each immediate form:
    - `ADC A,n` (`0xCE`)
    - `SUB n`   (`0xD6`)
    - `SBC A,n` (`0xDE`)
    - `XOR n`   (`0xEE`)
- Per-opcode result and flag semantics for at least one
  representative case in each family:
  - `ADC A,r` / `ADC A,(HL)` / `ADC A,n`: A ŌåÉ A + operand + C;
    S from bit 7, Z from zero, H from bit-3 carry, P/V from signed
    overflow, N=0, C from bit-7 carry.
  - `SUB r` / `SUB (HL)` / `SUB n`: A ŌåÉ A ŌłÆ operand; S from bit 7,
    Z from zero, H from bit-3 borrow, P/V from signed overflow,
    N=1, C from bit-7 borrow.
  - `SBC A,r` / `SBC A,(HL)` / `SBC A,n`: A ŌåÉ A ŌłÆ operand ŌłÆ C;
    flags as for `SUB` but accounting for the carry input.
  - `XOR r` / `XOR (HL)` / `XOR n`: A ŌåÉ A XOR operand; S from bit
    7, Z from zero, H=0, P/V from parity of result, N=0, C=0.
- For each `(HL)` form (`0x8E`, `0x96`, `0x9E`, `0xAE`) the test
  must exercise the memory read so the adapter's memory-read
  timing path is pinned, not only the ALU path.
- `PC` advance assertions: 1 byte for register and `(HL)` forms,
  2 bytes for each immediate form.
- Negative guard preserving the existing fail-fast contract for a
  still-out-of-scope sister opcode (for example one of the
  rotate/shift family bases or an ED-prefix opcode) so an
  accidental broadening of M46 scope is caught.
- Non-perturbation guard against an existing fixture ROM that does
  not exercise the newly covered behavior: frame, audio, and
  event-log digests must stay byte-identical before and after the
  change.
- Kickoff runtime coverage: the canonical PacManV8 T021 harness
  must advance past `PC=0x3EA` (`SUB n`) and the same-function
  `SUB B` sites at `PC=0x3BE`/`0x3CA`/`0x3D2` without aborting.
  Either the harness completes end-to-end, or it surfaces the next
  distinct blocker, which must then be recorded in the matching
  task file.

Exit criteria:
- Every opcode listed in Allowed scope is covered by the timed
  extracted core and by a dedicated focused test.
- The three redundant explicit entries (`0x91`, `0x93`, `0xAF`)
  have been removed from the `return 4` switch block, and the
  full test suite still passes ŌĆö i.e. the range-checks are
  demonstrably the single source of truth.
- `ctest --test-dir cmake-build-debug --output-on-failure` passes
  with the new coverage included, and no pre-existing regression is
  relaxed.
- Deterministic digests for unaffected fixture ROMs remain unchanged
  unless a deliberate change is explicitly recorded in the task file.
- The canonical PacManV8 T021 `tools/pattern_replay_tests.py`
  command either completes end-to-end against the current Vanguard8
  tree or stops on a new distinct blocker that is explicitly out of
  M46 scope, which is then recorded in `M46-T01` for the next
  milestone contract.
- The matching task file (`M46-T01`) captures the full opcode list,
  per-opcode test references, verification evidence, and ŌĆö if a new
  out-of-scope blocker is surfaced ŌĆö the precise next repro.

Forbidden extras:
- Turning this milestone into a broad extracted-core CPU completion
  pass by including rotate/shift, index-register, or ED-prefix
  families without a T021 repro.
- Touching the existing explicit `OR r` (`0xB0..0xB7`) or `CP r`
  (`0xB8..0xBF`) entries. The recommended fix explicitly says do
  **not** modify those ranges.
- Bundling unrelated real-ROM compatibility work not exercised by
  the PacManV8 T021 replay path.
- Editing PacManV8 sources to route around a Vanguard8 defect.
- Reintroducing the per-opcode milestone split for T021; a single
  follow-up milestone contract is required if a new out-of-scope
  gap is surfaced, not one per opcode.
