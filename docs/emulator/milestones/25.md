# Milestone 25 — Timed HD64180 VDP-B VRAM Seek OR n Compatibility (PacManV8)

Objective:
- Close the next narrow timed-CPU compatibility gap exposed by the PacManV8
  VDP-B maze render ROM: after the milestone-24 `LD A,B` patch, the extracted
  timed HD64180 runtime now aborts on `OR n` (`0xF6`) in the same VDP-B
  VRAM seek helper at `PC 0x0241`.

Required docs:
- `docs/spec/01-cpu.md`
- `docs/spec/02-video.md`
- `docs/spec/04-io.md`
- `docs/emulator/03-cpu-and-bus.md`
- `docs/emulator/07-implementation-plan.md`
- `docs/emulator/15-second-pass-audit.md`

Allowed scope:
- Timed opcode coverage inside the existing `src/core/cpu/` extracted-HD64180
  runtime path
- Tests and docs needed to lock the newly covered timed opcode surface
- Narrow runtime/integration coverage for the observed PacManV8 VDP-B VRAM
  seek blocker at `PC 0x0241`

Explicit non-goals:
- Full Z80/Z180 opcode completion (no broad logical-immediate or `OR r`
  sweep beyond `OR n`)
- New index-register, CB-prefixed, or block-instruction work unless the active
  failing path proves it is required
- VDP, renderer, scheduler, or audio changes unrelated to the timed CPU
  blocker
- Frontend or ROM-side work bundled into the same milestone
- Anything beyond the minimum needed to unblock the PacManV8 headless runtime
  frame dump at `PC 0x0241`

Allowed paths:
- `src/core/`
- `tests/`
- `docs/`

Required tests:
- Focused timed CPU coverage for `OR n` (`0xF6`) that pins the documented
  semantics (A <- A | n, PC advance by two, 7 T-states) and the Z80-standard
  flag result: S from bit 7 of the result, Z if result is zero, H cleared,
  P/V set for even parity, N cleared, C cleared
- Runtime/integration coverage reproducing the PacManV8 VDP-B VRAM seek
  blocker and proving execution survives the previously failing `PC 0x0241`

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 VDP-B VRAM seek
  path because timed execution is missing opcode `0xF6`
- The added timed opcode coverage is narrow, documented, and regression-covered
- Remaining opcode gaps stay explicit instead of expanding this milestone into
  a general CPU-completion pass

Forbidden extras:
- Turning the milestone into a broad CPU rewrite or generic logical-op table
- Bundling unrelated real-ROM compatibility fixes under the same task
- Loosening existing timed CPU or integration tests instead of adding the
  missing opcode coverage
- Adding VDP-B framebuffer, palette, or compositor changes to this milestone

Verification commands:
```bash
cmake -B build -DCMAKE_TOOLCHAIN_FILE=third_party/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build
ctest --test-dir build --output-on-failure
```
