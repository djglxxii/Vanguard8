# Milestone 37 — Timed HD64180 `LD E,n` Opcode Coverage for PacManV8 T020

Objective:
- Close the next real-ROM timed CPU compatibility gap exposed after M36
  unblocked the HALT-resume path. With foreground game-flow execution now
  resuming every VBlank, the PacManV8 blocked task
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T020-intermission-cutscenes.md`
  now reaches the intermission panel setup path and the canonical headless
  runtime aborts with:

  ```text
  Unsupported timed Z180 opcode 0x1E at PC 0x332E
  ```

- `PC=0x332E` is inside `intermission_fill_panel`'s scene-1 panel setup.
  The failing instruction is standard Z80 `LD E,n` (`E <- immediate byte`,
  `PC += 2`, 7 T-states, flags untouched). T020 genuinely needs it for the
  accepted rectangle-fill API (`BC=VRAM offset`, `D=height`, `E=width`,
  `A=fill byte`). The immediately preceding `LD D,n` (`0x16`) at `0x332C`
  executed without abort, so only the `0x1E` form is missing from the timed
  core.

- The same blocker record enumerates additional `LD E,n` sites along the
  active intermission drawing block: `0x330E`, `0x3317`, `0x3321`,
  `0x332E`, `0x3337`, `0x3341`, `0x3355`, `0x335E`, `0x3394`, `0x33A1`,
  `0x33AE`, `0x33BB`. All will exercise the same opcode handler.

- This milestone exists to close the `LD E,n` gap — and, to avoid another
  rebuild-per-opcode loop, the remaining missing `LD r,n` sister forms on
  the same immediate-byte family (`0x26` `LD H,n`, `0x2E` `LD L,n`) — in
  one documented pass, matching the family-closure precedent established
  by milestone 34. `0x06` / `0x0E` / `0x16` / `0x3E` are already timed.

Allowed scope:
- Timed opcode coverage inside the existing extracted HD64180 runtime for
  the missing `LD r,n` forms directly implicated by the T020 blocker:
  - `LD E,n` (`0x1E`) — the confirmed failing opcode at `PC 0x332E`.
  - `LD H,n` (`0x26`) — sister form, closed to prevent a follow-up
    blocker rebuild cycle on the same immediate-byte family.
  - `LD L,n` (`0x2E`) — sister form, same rationale.
- Implementation shape must follow the existing pattern in
  `third_party/z180/z180_core.cpp` for `op_ld_b_n` / `op_ld_c_n` /
  `op_ld_d_n` / `op_ld_a_n` (single `fetch_byte()` into the target
  register half; no flag updates).
- Adapter T-state timing entries for each opcode added in this milestone.
  Each `LD r,n` form costs **7 T-states** and must be registered in the
  existing 7-T-state case group in
  `src/core/cpu/z180_adapter.cpp` alongside the already-covered `0x06`,
  `0x0E`, `0x16`, `0x3E`.
- Tests and docs needed to lock the newly covered timed opcode surface.
- Narrow runtime/integration coverage proving the canonical PacManV8 T020
  minimal repro can run past `PC=0x332E` without a timed-opcode abort.

Explicit non-goals:
- **No general CPU opcode sweep.** No new arithmetic families,
  index-register (`DD`/`FD`) prefixes, `CB`-prefix work, or block
  opcodes.
- **No rewrite of existing `LD r,n` handlers.** `0x06` / `0x0E` /
  `0x16` / `0x3E` are already timed and must not be touched.
- **No interrupt-controller, scheduler, audio, VDP, or frontend
  changes.** This milestone is exclusively about adding three missing
  immediate-byte load opcodes and their 7-T-state timing entries.
- **No PacManV8 ROM edits.** The T020 evidence refresh happens in the
  PacManV8 repo after this milestone lands.
- **No M35 observability-flag changes.** `--inspect`, `--inspect-frame`,
  `--peek-mem`, `--peek-logical`, `--dump-cpu`, `--dump-vdp-regs`,
  `--dump-vram-a`, `--dump-vram-b`, and `--run-until-pc` must behave
  identically before and after this milestone.
- **No HALT-wake changes.** M36 owns that path; its tests must stay
  green unchanged.

Required tests:
- Focused timed CPU coverage in `tests/test_cpu.cpp` for each new
  opcode, pinning:
  - `LD E,n` (`0x1E`): after execution, `E == immediate byte`, `D`
    and all other registers unchanged, `PC` advanced by 2, flags
    (S/Z/H/PV/N/C) untouched, cycle count reports 7 T-states at the
    adapter boundary.
  - `LD H,n` (`0x26`): same contract with `H` as the destination.
  - `LD L,n` (`0x2E`): same contract with `L` as the destination.
- At least one test that places `LD D,n` (already covered) followed
  immediately by `LD E,n` and `LD A,n` to confirm the family-wide
  dispatch produces the correct register state after three sequential
  immediate-byte loads — this mirrors the exact PacManV8 pattern at
  `0x332C` (`LD D,0x44`), `0x332E` (`LD E,0x50`), `0x3330`
  (`LD A,0x22`).
- Adapter-level T-state coverage confirming each new opcode is
  classified in the 7-T-state case group and that the overall
  adapter dispatch remains deterministic.
- Non-perturbation guard: for a chosen ROM, `--frames`, and hashing
  selection that does not exercise the new opcodes, frame and audio
  digests before and after this milestone must be byte-identical.
  Reuse an existing M33 / M35-verified fixture already present under
  `tests/`.
- Runtime/integration coverage that rebuilds the PacManV8 ROM and
  proves the canonical T020 minimal repro
  `cmake-build-debug/src/vanguard8_headless --rom build/pacman.rom
  --frames 1020 --inspect-frame 1020 --dump-cpu --hash-frame 1020`
  completes without a timed-opcode abort at `PC=0x332E` (or at any of
  the sibling `LD E,n` sites listed in the blocker), with stable
  digest output across repeat runs.

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 T020 path
  with `Unsupported timed Z180 opcode 0x1E at PC 0x332E` (or at any of
  the sibling sites: `0x330E`, `0x3317`, `0x3321`, `0x3337`, `0x3341`,
  `0x3355`, `0x335E`, `0x3394`, `0x33A1`, `0x33AE`, `0x33BB`).
- The added timed opcode coverage for `LD E,n`, `LD H,n`, and
  `LD L,n` is narrow, documented, and regression-covered.
- `ctest --test-dir cmake-build-debug --output-on-failure` passes with
  the new opcode tests included, and no pre-existing CPU test is
  relaxed or removed.
- Deterministic digests for ROMs that do not depend on the new opcodes
  are unchanged.
- PacManV8 T020 becomes actionable again from the emulator side; the
  actual cutscene evidence refresh and acceptance still happen in the
  PacManV8 repo.

Forbidden extras:
- Turning this milestone into another broad general opcode sweep.
- Bundling unrelated PacManV8 compatibility fixes under the same task.
- Loosening existing timed CPU or integration tests instead of adding
  the missing opcode coverage alongside them.
- Altering PacManV8 sources to route around the blocker.
- Adding index-register (`DD`/`FD`) `LD r,n` variants — those are not
  on the T020 path and would expand scope beyond the confirmed need.
