# Milestone 35 — Agent Observability Primitives for Headless ROM Verification

Objective:
- Close the observability gap that has now re-blocked PacManV8 T020 a second
  time. M34 fixed a CPU timed-opcode gap that kept the T020 intermission path
  from executing; the follow-up runtime verification revealed that
  `vanguard8_headless` cannot distinguish between
  "state machine never reached the intermission owner",
  "reached it but crashed before drawing",
  and "reached it and drew, but the draw is not visible in the composed
  framebuffer." The canonical T020 blocker note records identical composed
  frame hashes at frames `1020`, `1770`, `2520`, and `2640` across three
  different intermission render strategies, which is only useful as a negative
  result — it does not identify which of the three failure modes is in play.
- Add headless CLI primitives that surface the existing `Bus`,
  `video::V9938`, and `cpu::Z180Adapter` const/snapshot accessors as
  deterministic introspection output. After this milestone, a coding agent
  running `vanguard8_headless` must be able to answer:
  - Did the CPU actually reach a given PC during the run?
  - What is the value of a known RAM address at the end of frame N?
  - What did VDP-A / VDP-B VRAM, registers, palette, and status contain at the
    end of frame N?
  - What are the CPU registers, flags, halt state, and MMU mapping at the end
    of frame N?
- This milestone is deliberately scoped to the headless CLI surface and its
  tests. The underlying core inspection APIs already exist; what is missing is
  a supported, deterministic way for agents to consume them.

Explicit non-goals:
- **No core-behavior changes.** No new tracking buffers, no new instrumentation
  hooks in the scheduler, audio, VDP, CPU adapter, or `Bus`, beyond what is
  already exposed as `const` / snapshot accessors. If an inspection flag would
  require adding new state to core, the flag is out of scope for this
  milestone and must be deferred.
- **No determinism changes.** Any new flag must not perturb CPU execution,
  event timing, audio sample counts, frame hashes, audio hashes, event-log
  digests, or save-state compatibility. Adding a flag must be observably a
  no-op on every existing deterministic guard.
- **No GUI / ImGui / frontend playback work.** The desktop debugger panels
  described in `docs/emulator/06-debugger.md` remain future work.
- **No opcode, VDP, audio, scheduler, cartridge, input, or save-state feature
  work.** If an observed gap needs those, record it and file a separate
  milestone.
- **No PacManV8 ROM edits.** The T020 evidence refresh happens in the
  PacManV8 repo after this milestone lands.
- **No new binaries.** Everything is inside the existing `vanguard8_headless`
  binary. The `vanguard8_frontend` binary is not in scope.

Required CLI surface:
- `--peek-mem ADDR[:LEN]` (repeatable)
  - Physical-address byte dump via `core::Bus::read_memory`.
  - `ADDR` is a hex physical address (no `0x` prefix required, both accepted).
    `LEN` defaults to `1`, clamped to a small upper bound (for example, 256)
    per invocation.
  - Reads are performed at the end of the selected inspection frame
    (see `--inspect-frame`), never mid-instruction.
- `--peek-logical ADDR[:LEN]` (repeatable)
  - Same as `--peek-mem` but translated through the current MMU state
    (`CBAR`/`CBR`/`BBR`) at inspection time. Prints both the logical and the
    resolved physical address alongside the bytes.
  - If the logical range crosses an MMU region boundary, the report must
    either split the dump per region or label the boundary — the exact choice
    is up to the task, but the output must be deterministic and documented.
- `--dump-vram-a PATH` / `--dump-vram-b PATH`
  - Write the full 64 KB VRAM contents of the selected VDP at the inspection
    frame as a raw binary file. Paired with the stdout digest so agents can
    regression-hash without reading the file.
- `--dump-vdp-regs`
  - Emit all V9938 register values (R#0..R#46), status (S#0..S#9), palette
    (both raw 9-bit entries and decoded RGB), command-state snapshot, and
    `current_visible_width` for both VDPs into the inspection report.
- `--dump-cpu`
  - Emit `PC`, `A`, `BC`, `DE`, `HL`, `AF'`/`BC'`/`DE'`/`HL'`, `IX`, `IY`,
    `SP`, `I`, `R`, `IFF1`/`IFF2`, halt state, `CBAR`, `CBR`, `BBR`, and the
    decoded active bank into the inspection report. Must draw only from
    `cpu::Z180Adapter::state_snapshot` / existing public accessors.
- `--inspect-frame N`
  - Pin all of the above introspection to the end of frame `N`. Defaults to
    the last completed frame if omitted. Must coexist with `--hash-frame`
    without altering hash output.
- `--inspect PATH`
  - Write a single deterministic inspection-report text file at `PATH`
    containing all active `--dump-cpu`, `--dump-vdp-regs`, `--peek-mem`,
    and `--peek-logical` results for the selected inspection frame. The
    exact format is locked by a golden test in `tests/`.
  - If `--inspect` is given without explicit dump flags, the report includes
    CPU + both VDPs' register blocks by default (no memory peeks unless
    requested).
  - When `--inspect` is omitted, any `--peek-mem` / `--peek-logical` /
    `--dump-cpu` / `--dump-vdp-regs` flags emit to stdout under a fixed
    section header so stdout-only agents still get a deterministic snapshot.
- `--run-until-pc HEX[:MAX_FRAMES]`
  - Drive the existing `cpu::Z180Adapter::add_breakpoint`
    (`BreakpointType::pc`) from the CLI. Run up to `MAX_FRAMES` frames; if
    the breakpoint fires, stop at end-of-frame of the frame in which it fired
    and record the hit frame index and master-cycle in the final report.
    If it never fires, the report records "not-hit" with the final frame
    count. `MAX_FRAMES` defaults to the existing `--frames` value when
    omitted.
  - This flag may be combined with `--inspect`; the inspection frame becomes
    the stop frame when a hit occurs, or `--inspect-frame` when provided.
- `--help`
  - Must be updated to list every new flag with a one-line usage string.

Explicit format requirements:
- Every new piece of output must be deterministic across repeat runs of the
  same ROM/input. No timestamps, no process IDs, no absolute host paths in
  the body of the inspection report (paths may appear in a clearly-marked
  "invocation" header that the golden test ignores, or the report must use
  only the file basename — either choice is acceptable but must be
  consistent and covered by tests).
- Byte, register, and address values are rendered as lowercase hex.
- Numeric fields (frame index, master cycle, CPU T-state count) are rendered
  as decimal.
- The existing `--hash-frame` / `--expect-frame-hash` / `--hash-audio` /
  `--expect-audio-hash` / `--dump-frame` / `--dump-fixture` / `--trace`
  paths must behave identically with or without the new flags (verified by
  test).

Required tests:
- A headless-CLI test that invokes `vanguard8_headless --rom <fixture>
  --frames N --inspect <tmp>` against an existing deterministic fixture ROM
  and diffs the emitted report against a committed golden file.
- A determinism guard: running the same command twice produces byte-identical
  inspection reports and byte-identical VRAM dumps.
- A non-perturbation guard: for a chosen `--rom`, `--frames` value, and
  `--hash-frame` / `--hash-audio` selection, adding `--inspect`,
  `--peek-mem`, `--peek-logical`, `--dump-cpu`, `--dump-vdp-regs`,
  `--dump-vram-a`, `--dump-vram-b`, and `--run-until-pc` (individually and
  together, with an unreachable PC target for the combined case) produces
  identical frame digests, audio digests, and event-log digests relative to
  the non-inspecting invocation.
- Focused coverage for `--peek-mem` and `--peek-logical` using a ROM with
  known-initialized SRAM bytes; the test must verify that the printed bytes
  match `core::Bus::read_memory` called directly at the same frame.
- Focused coverage for `--dump-vram-a` / `--dump-vram-b` proving the dumped
  bytes match `video::V9938::vram()` at the chosen frame and that the
  dump's reported digest matches a separately-computed SHA-256 of the file.
- Focused coverage for `--run-until-pc` proving both the "hit inside frame
  budget" and "not hit inside frame budget" paths report the expected frame
  index and exit code, with no core behavior drift when the flag is absent.
- A smoke-level PacManV8 integration regression:
  ```bash
  cmake-build-debug/src/vanguard8_headless \
      --rom /home/djglxxii/src/PacManV8/build/pacman.rom \
      --frames 120 --inspect-frame 120 --inspect /tmp/pacman-m35.txt \
      --peek-mem 0xF8250:16 --peek-mem 0xF8270:8 --dump-cpu --dump-vdp-regs
  ```
  locked against a committed golden digest of `/tmp/pacman-m35.txt`
  (the peek addresses cover `GAME_FLOW_STATE_BASE = 0x8250` and
  `INTERMISSION_STATE_BASE = 0x8270` mapped to physical SRAM at `0xF0000 +`
  offset via the boot-time `CBAR=0x48 / CBR=0xF0`). The digest itself is
  filled in by the implementing task; the requirement here is only that the
  golden exists, is reproducible, and is captured in the task's completion
  summary.

Exit criteria:
- `vanguard8_headless --help` documents every flag defined in "Required CLI
  surface".
- `ctest --test-dir cmake-build-debug --output-on-failure` passes, with the
  new headless-CLI tests included.
- The PacManV8 smoke regression above produces a deterministic, non-empty
  inspection report that includes CPU state, both VDP register blocks, the
  two peeked RAM ranges, and no host-specific content.
- `docs/emulator/06-debugger.md` contains a "Headless Observability" section
  describing every new flag, its arguments, its output, and the intended
  agent workflow (for example, "to distinguish state-never-reached from
  state-reached-but-crashed, use `--run-until-pc` on the entry symbol and
  `--peek-mem` on the relevant state base").
- A PacManV8 agent can take the T020 minimal repro, re-apply the T020 WIP
  wiring, and use only the new headless flags to decide between the three
  failure modes recorded in the T020 blocker note — without touching any
  GUI tooling and without requiring new emulator-side work.

Forbidden extras:
- Adding live debugger panels, ImGui UI, or any rendering-side change.
- Adding new core-state tracking (new buffers, new counters, new recorded
  events) to support the new flags. If a flag would require it, defer to a
  later milestone.
- Loosening existing deterministic tests. New tests must be added alongside
  existing ones, not replace them.
- Bundling unrelated emulator fixes (opcode work, audio, VDP, scheduler,
  save-state) into this milestone.
- Changing the semantics of existing flags (`--frames`, `--hash-frame`,
  `--expect-frame-hash`, `--hash-audio`, `--expect-audio-hash`,
  `--dump-frame`, `--dump-fixture`, `--trace`, `--trace-instructions`,
  `--symbols`, `--press-key`, `--gamepad1-button`, `--gamepad2-button`,
  `--replay`, `--recent`, `--paused`, `--step-frame`, `--vclk`).
