# Milestone 36 — Fix HD64180 HALT Resume After Interrupt

Objective:
- Close the runtime CPU defect that has re-blocked PacManV8 T020 after
  milestone 35 gave the blocker enough observability to pinpoint the cause.
  The canonical blocker record
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T020-intermission-cutscenes.md`
  now reports, using the M35 flags:
  - CPU is stuck at `PC=0x0067`, `halted=true`, `IFF1=true`, `IFF2=true` out to
    frame `1100`.
  - `0x0067` is `idle_loop`'s `HALT`; the instruction after `HALT` at
    `0x0068` is `call game_flow_update_frame`, which is never reached
    (`--run-until-pc 0x0068:20` reports `not-hit`).
  - `GAME_FLOW_FRAME_COUNTER=0x0000` stays at zero at frame `1100`, while
    `AUDIO_FRAME_COUNTER=0x0437` continues to advance, so the interrupt
    service path is executing but the foreground `HALT`-resume path is not.
  - `intermission_start` at `0x2EE3` is therefore never reached, which is why
    T020 could not observe post-handoff frame divergence under three
    different intermission render strategies.
- The defect is in the canonical HD64180 runtime HALT semantics. `op_halt`
  decrements `pc_` back to the HALT instruction and sets `halted_ = true`;
  `service_pending_interrupt` and `service_vectored_interrupt` then
  `push_word(pc_.value)` — which pushes the HALT instruction's own address
  as the interrupt return target. After `RETI`, the CPU returns to the
  HALT instruction, re-enters halt, and the foreground frame loop after
  `HALT` is never executed. INT1 / audio still visibly advances because the
  ISR itself runs each time the interrupt fires; the problem is purely the
  resume-after-HALT return address.
- This milestone fixes that resume path so that, when an interrupt is
  accepted while the CPU is halted, `halted_` is cleared and the interrupt
  return address recorded on the stack is the instruction *after* `HALT`
  (the accepted Z80/HD64180 semantics). Once fixed, the documented T018
  frame-loop contract (`HALT` then `call game_flow_update_frame`) resumes
  every VBlank and the existing T020 wiring can produce distinct frame
  evidence instead of a re-halt loop.

Allowed scope:
- Fix the HALT-resume contract inside the extracted HD64180 runtime
  (`third_party/z180/z180_core.{hpp,cpp}`) so that after any accepted
  interrupt (INT0 IM1, INT1 vectored IM2, PRT0, PRT1):
  - `halted_` is cleared before the handler runs, and
  - the address pushed on the stack is the instruction after the `HALT`
    opcode, not the `HALT` address itself.
  The implementor may choose any of the documented Z80/HD64180 realizations
  (for example, not decrementing `pc_` in `op_halt` and instead re-fetching
  the HALT opcode each step while `halted_` is set, or keeping the
  `pc_ -= 1` convention in `op_halt` and bumping `pc_ += 1` at the top of
  interrupt servicing before `push_word`). The choice must be consistent
  across every interrupt source and must be spec-traceable.
- Adapter-side integration in `src/core/cpu/z180_adapter.cpp` only as
  needed to preserve correct T-state accounting for HALT-wake and
  interrupt-service paths after the fix. No new public state.
- Focused extracted-core CPU tests that pin:
  - HALT with interrupts disabled stays halted indefinitely and leaves
    `pc_` pointing at the HALT opcode as seen by snapshots.
  - HALT with INT0 asserted and `IFF1=1`, `IM=1` wakes the CPU, pushes the
    address of the instruction after HALT, jumps to `0x0038`, and on
    `RETI` resumes at the instruction after HALT (not at HALT).
  - HALT with INT1 asserted and `ITE1=1`, `IFF1=1`, `IM=2` wakes the CPU,
    uses the vectored handler address, pushes the address of the
    instruction after HALT, and on `RETI` resumes at the instruction
    after HALT.
  - HALT with PRT0/PRT1 timer interrupts pending produces the same
    resume semantics via their existing `service_vectored_interrupt`
    path.
  - `IFF1`/`IFF2` transitions across the HALT-wake path match the
    accepted Z80/HD64180 convention already used by non-halted
    interrupt servicing.
- Narrow runtime/integration coverage proving the canonical PacManV8 T020
  minimal repro now reaches `0x0068` early and `0x2EE3` by the documented
  handoff frame.

Explicit non-goals:
- **No general CPU opcode sweep.** No new arithmetic families,
  index-register prefixes, `CB`-prefix work, or block opcodes.
- **No interrupt-controller rewiring.** INT0 / INT1 / PRT0 / PRT1
  acceptance conditions stay as written in `docs/spec/01-cpu.md` and
  `docs/spec/04-io.md`. The fix is exclusively about HALT-wake resume
  semantics, not about changing when an interrupt is accepted.
- **No scheduler, audio, VDP, or frontend changes.** The scheduler's
  `cpu_.halted() && !int0 && !int1` early-exit must still work; this
  milestone must not perturb audio hashes, frame hashes, or event-log
  digests for ROMs that do not depend on HALT-wake resume.
- **No PacManV8 ROM edits.** The T020 evidence refresh happens in the
  PacManV8 repo after this milestone lands.
- **No M35 observability-flag changes.** `--inspect`, `--inspect-frame`,
  `--peek-mem`, `--peek-logical`, `--dump-cpu`, `--dump-vdp-regs`,
  `--dump-vram-a`, `--dump-vram-b`, and `--run-until-pc` must behave
  identically before and after this milestone.

Observability note on `--run-until-pc` for this fix:
- `--run-until-pc 0x0068:20` will continue to report `not-hit` after the
  fix, because the M35 CLI samples PC only at end-of-frame boundaries
  and `0x0068` is traversed mid-frame (the CPU resumes at `0x0068` on
  VBlank, runs through `call game_flow_update_frame`, and returns to
  `HALT` at `0x0067` before the frame boundary is checked). This is a
  known limitation of the M35 sampling design, not a defect of this
  fix. The authoritative post-fix evidence is the peek-logical output
  above showing game-flow state bytes advance past zero.
- `--run-until-pc 0x0067:N` is non-discriminating: the CPU rests at
  `0x0067` at every frame boundary both before and after the fix.
- Closing the mid-frame PC sampling gap is deferred to a future M35+
  observability milestone; it is explicitly out of scope here.
