# Milestone 26 — Timed HD64180 VDP-B Framebuffer Load LD DE,nn Compatibility (PacManV8)

Objective:
- Close the next narrow timed-CPU compatibility gap exposed by the PacManV8
  VDP-B maze render ROM: after the milestone-25 `OR n` patch, the extracted
  timed HD64180 runtime now aborts on `LD DE,nn` (`0x11`) in the VDP-B
  framebuffer-load setup at `PC 0x020B`.

Required docs:
- `docs/spec/01-cpu.md`
- `docs/spec/02-video.md`
- `docs/spec/04-io.md`
- `docs/emulator/03-cpu-and-bus.md`
- `docs/emulator/07-implementation-plan.md`
- `docs/emulator/15-second-pass-audit.md`

Allowed scope:
- Timed opcode coverage inside the existing `src/core/cpu/` extracted-HD64180
  runtime path
- Tests and docs needed to lock the newly covered timed opcode surface
- Narrow runtime/integration coverage for the observed PacManV8 VDP-B
  framebuffer-load blocker at `PC 0x020B`

Explicit non-goals:
- Full Z80/Z180 opcode completion (no broad `LD rr,nn` sweep beyond
  `LD DE,nn`; `LD BC,nn`, `LD HL,nn`, and `LD SP,nn` are already covered
  by earlier milestones)
- New index-register, CB-prefixed, or block-instruction work unless the active
  failing path proves it is required
- VDP, renderer, scheduler, or audio changes unrelated to the timed CPU
  blocker
- Frontend or ROM-side work bundled into the same milestone
- Anything beyond the minimum needed to unblock the PacManV8 headless runtime
  frame dump at `PC 0x020B`

Allowed paths:
- `src/core/`
- `tests/`
- `docs/`

Required tests:
- Focused timed CPU coverage for `LD DE,nn` (`0x11`) that pins the documented
  16-bit immediate load semantics, the 10 T-state timing, the three-byte PC
  advance, and that flags are left untouched
- Runtime/integration coverage reproducing the PacManV8 VDP-B framebuffer-load
  blocker and proving execution survives the previously failing `PC 0x020B`

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 VDP-B
  framebuffer-load path because timed execution is missing opcode `0x11`
- The added timed opcode coverage is narrow, documented, and regression-covered
- Remaining opcode gaps stay explicit instead of expanding this milestone into
  a general CPU-completion pass

Forbidden extras:
- Turning the milestone into a broad CPU rewrite or generic `LD rr,nn` table
- Bundling unrelated real-ROM compatibility fixes under the same task
- Loosening existing timed CPU or integration tests instead of adding the
  missing opcode coverage
- Adding VDP-B framebuffer, palette, or compositor changes to this milestone

Verification commands:
```bash
cmake -B build -DCMAKE_TOOLCHAIN_FILE=third_party/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build
ctest --test-dir build --output-on-failure
```
