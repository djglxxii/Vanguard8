# Vanguard 8 Emulator — 1.0 Readiness Snapshot

## Purpose

This note is the milestone-11 readiness surface used to judge whether the repo
can be reviewed for final acceptance from repository state alone.

It summarizes:

- what is implemented and regression-covered
- which limitations are still intentional

## Implemented and Locked Surfaces

- HD64180 boot MMU mapping, bank switching, interrupt dispatch, DMA, and PRT
  timer surfaces covered by unit tests
- dual-V9938 compositing, covered Graphic 3 and Graphic 4 rendering paths,
  Mode 2 sprite surface, and covered VDP command timing
- YM2151, AY-3-8910, MSM5205, mixer/resampler, and scheduler-driven audio
  timing
- controller input, ROM loading, debugger panels, save states, rewind, replay,
  and headless regression

## Remaining Intentional Limitations

These are still documented limitations, not accidental gaps:

- horizontal scroll via `R#26`/`R#27` remains deferred pending source-backed
  spec verification
- Graphic 3 LN=1 background fetch for lines `192-211` is intentionally locked
  to backdrop output until the repo spec defines the fetch rule explicitly
- Graphic 5 palette mapping is not implemented because the repo spec still says
  it must be verified against the V9938 databook
- Graphic 7 direct-color encoding is not implemented because the repo spec
  still says it must be verified against the V9938 databook
- 4-player expansion and battery-backed cartridge SRAM mapping details remain
  deferred by spec
- the repo proves a tested HD64180 execution subset, but does not claim a full
  audited equivalence between the extracted MAME Z180 core and all HD64180
  behavior; see `docs/emulator/08-compatibility-audit.md`
- no external game-ROM corpus is checked into the repo, so broader real-ROM
  compatibility remains a documented out-of-repo validation activity

## Review Posture

The repo is closest to 1.0 readiness where behavior is explicitly specified and
regression-covered. Acceptance review should focus on:

- whether the remaining limitations above are narrow enough to be acceptable
- whether any of those limitations must be promoted into stronger spec text
- whether milestone task files and verification evidence are complete
