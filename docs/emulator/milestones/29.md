# Milestone 29 — Timed HD64180 VDP-B Framebuffer Copy RET Z / DEC DE Compatibility (PacManV8)

Objective:
- Close the remaining narrow timed-CPU compatibility gaps designated by the
  PacManV8 VDP-B maze render blocked task after the milestone-28 `OR E`
  patch. The blocked task's static audit (see
  `/home/djglxxii/src/PacManV8/docs/tasks/blocked/T007-vdp-b-maze-render-and-pellet-display.md`)
  identifies exactly two remaining missing timed opcodes on the VDP-B
  framebuffer copy loop path:
  - `0xC8` at `PC 0x0248` — `RET Z`
  - `0x1B` at `PC 0x024D` — `DEC DE`
- These are the only opcodes authorized by the T007 blocker record for this
  milestone. Implement both together so the documented PacManV8 headless
  runtime frame dump path can clear the known gaps in a single pass and
  T007 can be unblocked.

Required docs:
- `docs/spec/01-cpu.md`
- `docs/spec/02-video.md`
- `docs/spec/04-io.md`
- `docs/emulator/03-cpu-and-bus.md`
- `docs/emulator/07-implementation-plan.md`
- `docs/emulator/15-second-pass-audit.md`

Allowed scope:
- Timed opcode coverage inside the existing `src/core/cpu/` extracted-HD64180
  runtime path for `RET Z` (`0xC8`) and `DEC DE` (`0x1B`) only
- Tests and docs needed to lock the newly covered timed opcode surface
- Narrow runtime/integration coverage for the observed PacManV8 VDP-B
  framebuffer copy blockers at `PC 0x0248` and `PC 0x024D`

Explicit non-goals:
- Full Z80/Z180 opcode completion (no broad conditional-return sweep beyond
  `RET Z`; no broad 16-bit `DEC rr` sweep beyond `DEC DE`)
- New index-register, CB-prefixed, or block-instruction work unless the
  active failing path proves it is required
- VDP, renderer, scheduler, or audio changes unrelated to the timed CPU
  blockers
- Frontend or ROM-side work bundled into the same milestone
- Anything beyond the minimum needed to unblock the PacManV8 headless
  runtime frame dump at `PC 0x0248` and `PC 0x024D`
- Opening a broader "general opcode audit" effort inside this milestone;
  any additional gaps surfaced by the full ROM run after M29 belong to a
  future narrow compatibility milestone

Allowed paths:
- `src/core/`
- `tests/`
- `docs/`

Required tests:
- Focused timed CPU coverage for `RET Z` (`0xC8`) that pins the documented
  conditional-return semantics: when Z=1, PC <- (SP), (SP+1); SP advances
  by two; timing is 11 T-states on the taken path and 5 T-states on the
  not-taken path; flags are left untouched
- Focused timed CPU coverage for `DEC DE` (`0x1B`) that pins the documented
  16-bit decrement semantics: DE <- DE - 1 (wrapping at 0x0000); PC
  advances by one; 6 T-states; flags are left untouched
- Runtime/integration coverage reproducing the PacManV8 VDP-B framebuffer
  copy blockers and proving execution survives the previously failing
  `PC 0x0248` (`RET Z`) and `PC 0x024D` (`DEC DE`)

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 VDP-B
  framebuffer copy path because timed execution is missing `0xC8` or
  `0x1B`
- The added timed opcode coverage is narrow, documented, and
  regression-covered
- Remaining opcode gaps (beyond the two designated by the blocked T007
  task) stay explicit and deferred, rather than expanding this milestone
  into a general CPU-completion pass
- The PacManV8 T007 task becomes actionable for runtime frame capture
  against `vanguard8_headless` (unblocking confirmation happens in the
  PacManV8 repo, not here)

Forbidden extras:
- Turning the milestone into a broad CPU rewrite or generic conditional-return
  / 16-bit-decrement table
- Bundling unrelated real-ROM compatibility fixes under the same task
- Loosening existing timed CPU or integration tests instead of adding the
  missing opcode coverage
- Adding VDP-B framebuffer, palette, or compositor changes to this milestone

Verification commands:
```bash
cmake -B build -DCMAKE_TOOLCHAIN_FILE=third_party/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build
ctest --test-dir build --output-on-failure
```
