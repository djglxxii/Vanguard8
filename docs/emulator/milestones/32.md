# Milestone 32 — Frontend Live Audio Playback for Interactive Verification

Objective:
- Close the gap between the deterministic core audio pipeline and the
  interactive desktop frontend so that running a ROM in
  `cmake-build-debug/src/vanguard8_frontend` produces audible output from
  the YM2151, AY-3-8910, and MSM5205 chips. Today the PacManV8 T017 FM
  music cues, T016 PSG effects, and any future audio-producing ROM are
  inaudible in the GUI, which blocks human review by ear. The
  deterministic headless audio hash path (`vanguard8_headless
  --hash-audio`) must continue to produce identical digests after this
  milestone lands.
- The frontend already contains scaffolding (`SdlAudioOutputDevice`,
  `AudioQueuePump`, `app.cpp` open/pump/close lifecycle, pumping
  `AudioMixer::consume_output_bytes()` per frame), but in practice
  launching a ROM yields silence. This milestone is to diagnose the
  silent-output path, fix it, lock it with tests, and bring the audio
  design doc back in sync with the shipped implementation.

Required docs:
- `docs/spec/03-audio.md`
- `docs/emulator/05-audio.md` (will be updated by this milestone)
- `docs/emulator/07-implementation-plan.md` (Milestone 12 audio-output
  exit criterion: "audible audio")
- `/home/djglxxii/src/PacManV8/docs/tasks/completed/T016-psg-sound-effects.md`
- `/home/djglxxii/src/PacManV8/docs/tasks/active/T017-fm-music.md`

Allowed scope:
- Investigation of the current frontend audio path and identification of
  the concrete cause(s) of silent output (e.g., mixer not producing
  output bytes under the GUI scheduler path, sample-rate/format
  mismatch, device opened with wrong parameters, queue never draining,
  pump never invoked under the live run loop, `SDL_PauseAudioDevice`
  ordering, master-volume or mute in the chosen backend, etc.).
- Code fixes inside `src/frontend/` to make audio audible when a ROM
  writes to any of the three audio chips, using the existing
  `AudioMixer` deterministic byte stream as the source of truth. No new
  resamplers or mixing stages may be added inside `src/core/`; the
  frontend consumes whatever the core already produces.
- Minimal, surgical adjustments inside `src/core/audio/` only if
  investigation shows the mixer is not actually populating
  `output_bytes_` under the frontend run-loop path (e.g., a gating
  condition that was intended for headless-only). Any such change must
  preserve the headless audio SHA-256 digests currently produced for:
  - PacManV8 `pacman.rom` at `--frames 300 --hash-audio` →
    `0d634fb059d15b12c4a8faf2412fbe08b85d187a31b1f22278ce3662f3b44390`
  - All other existing audio-hash-gated tests in `tests/`.
- Frontend CLI / configuration additions strictly limited to audio
  controls needed for bring-up: a `--no-audio` / `--mute` flag, optional
  sample-rate override, and an optional audio backend-device selection.
  These are nice-to-haves; the milestone does not require them, but if
  added they must be documented and tested.
- Regression surface for the live-audio path:
  - Unit/integration tests for `AudioQueuePump` and
    `SdlAudioOutputDevice` fake-device behavior covering startup,
    shutdown, empty-input, queue-full back-pressure, and reopen.
  - A parity test that runs the PacManV8 review schedule (or an
    equivalent synthetic ROM) through both the headless and frontend
    mixer consumption paths and confirms the produced PCM byte stream
    is bit-identical, so that what is played through speakers is
    exactly what the deterministic digest is over.
- Documentation updates:
  - Rewrite the "Mixer And Resampler" section of
    `docs/emulator/05-audio.md` to remove the "does not yet allocate an
    SDL ring buffer" note and describe the shipping GUI audio path.
  - Add a short "Interactive audio verification" subsection describing
    how to use the GUI frontend to audibly check a ROM, and the
    `--no-audio` flag if it is introduced.

Explicit non-goals:
- Any change to the deterministic core audio chain: no new resampler,
  no new mixing math, no changes to `Ym2151`, `Ay8910`, or the
  MSM5205 adapter's sample production. If a chip-side bug is suspected,
  it belongs to its own milestone.
- Expanding the frontend beyond audio playback: no new debugger UI, no
  video changes, no input changes.
- ImGui/Dear-ImGui style audio visualization, spectrum displays, volume
  sliders beyond a simple mute toggle, or per-chip balance controls.
- Replacing SDL with a different audio backend. If SDL is
  insufficient, raise a follow-up milestone; this one stays on SDL.
- Adding audio recording/export features, WAV dumps, or loopback
  testing (beyond what is needed for the parity test above).
- Changing `vanguard8_headless` audio behavior, flags, or digests.
- Re-opening any opcode, CPU, VDP, palette, renderer, or scheduler
  work. Audio-side only.

Allowed paths:
- `src/frontend/`
- `src/core/audio/` (only for the gated case described above)
- `tests/`
- `docs/emulator/05-audio.md`
- `docs/emulator/current-milestone.md` (only when activating/retiring
  this milestone)
- `docs/emulator/milestones/32.md` (this file, for status updates)

Required tests:
- Fake-device coverage for `SdlAudioOutputDevice`'s public contract
  (`open` / `queue_audio` / `queued_bytes` / `close`) including the
  error-return paths and the re-open-after-close case.
- `AudioQueuePump` coverage for: empty input pass-through, back-pressure
  when `queued_bytes() >= max_queue_bytes_`, and successful forwarding
  of non-empty byte vectors.
- Frontend runtime-lifecycle test that, using a fake audio device,
  proves: device is opened before the first frame is pumped, the pump
  is invoked every frame while the runtime is running, and the device
  is closed on shutdown.
- Headless/frontend PCM-parity test: given a fixed ROM and frame count,
  the concatenation of `AudioMixer::consume_output_bytes()` returned on
  each frame is byte-identical between the headless runner and the
  frontend run loop with a fake audio device that records everything
  it is asked to queue.
- Headless digest regression: the existing audio-hash-gated ctest cases,
  plus an explicit check that
  `vanguard8_headless --rom <PacManV8 pacman.rom> --frames 300
   --hash-audio` still produces
  `0d634fb059d15b12c4a8faf2412fbe08b85d187a31b1f22278ce3662f3b44390`
  after this milestone's changes.

Exit criteria:
- Launching
  `cmake-build-debug/src/vanguard8_frontend /path/to/pacman.rom`
  produces audible output from the host speakers for the PacManV8
  T017 review schedule: the T016 PSG cues at frames 0/12/36/72/112
  and the T017 FM cues at frames 144/196/240 can all be heard without
  crashes, stalls, or audio dropouts in a 300-frame window, under a
  normal desktop SDL audio configuration.
- The same binary continues to run without regression for ROMs that
  do not write to the audio chips (no spurious noise from an unfed
  device).
- `ctest --test-dir cmake-build-debug --output-on-failure` is green,
  with the new tests listed under "Required tests" included.
- `docs/emulator/05-audio.md` no longer says the SDL ring buffer has
  not been allocated; the shipping audio path is documented.
- Running the canonical headless PacManV8 verification command (below)
  continues to produce the T017 audio SHA-256 digest unchanged.

Forbidden extras:
- Bundling video, input, VDP, or debugger-UI work into this milestone.
- Adding audio features that are not needed to make ROM audio audible
  (EQ, per-chip balance, spectrum visualizers, recording, virtual
  loopback).
- Introducing a second audio mixing/resampling path in the frontend.
  There is exactly one deterministic mixer in `src/core/audio/`; the
  frontend only consumes its bytes.
- Relaxing any existing audio-hash-gated test to accommodate a
  frontend change. If a core change breaks a digest, the milestone
  must either back it out or raise a separate, narrowly-scoped
  compatibility milestone.
- Changing `vanguard8_headless` CLI, flags, or exit codes.

Verification commands:
```bash
cmake --build cmake-build-debug
ctest --test-dir cmake-build-debug --output-on-failure

# Determinism regression guard (must remain unchanged):
cmake-build-debug/src/vanguard8_headless \
    --rom /home/djglxxii/src/PacManV8/build/pacman.rom \
    --frames 300 --hash-audio \
    --expect-audio-hash \
    0d634fb059d15b12c4a8faf2412fbe08b85d187a31b1f22278ce3662f3b44390

# Interactive listening check (human verification):
cmake-build-debug/src/vanguard8_frontend \
    /home/djglxxii/src/PacManV8/build/pacman.rom
```
