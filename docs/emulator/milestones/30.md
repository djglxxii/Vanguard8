# Milestone 30 — Timed HD64180 PacManV8 VBlank Handler PUSH/POP rr Compatibility

Objective:
- Close the narrow timed-CPU compatibility gap blocking the PacManV8 T016
  audio bring-up ROM against the current Vanguard8 timed HD64180 runtime.
  The PacManV8 VBlank handler (see
  `/home/djglxxii/src/PacManV8/docs/field-manual/vanguard8-build-directory-skew-and-timed-opcodes.md`)
  was widened to save more register pairs around `audio_update_frame`, so the
  timed dispatcher now observes the `PUSH BC/DE/HL` and `POP BC/DE/HL`
  register-pair stack opcodes that the timed path does not yet cover. The
  first observed abort is:
  - `0xC5` (`PUSH BC`) at `PC 0x0039`
- The same handler then exercises the rest of the register-pair stack family:
  - `0xD5` (`PUSH DE`) at `PC 0x003A`
  - `0xE5` (`PUSH HL`) at `PC 0x003B`
  - `0xE1` (`POP HL`) at `PC 0x0040`
  - `0xD1` (`POP DE`) at `PC 0x0041`
  - `0xC1` (`POP BC`) at `PC 0x0042`
- These six opcodes are one structurally homogeneous family. The field manual
  explicitly recommends closing them together rather than one-opcode-at-a-time
  to avoid a repair loop on every VBlank rebuild. Implement all six in one pass
  so the PacManV8 T016 regression command can clear the interrupt handler
  prologue and epilogue without further timed-path aborts.

Required docs:
- `docs/spec/01-cpu.md`
- `docs/spec/04-io.md`
- `docs/emulator/03-cpu-and-bus.md`
- `docs/emulator/07-implementation-plan.md`
- `docs/emulator/15-second-pass-audit.md`
- `/home/djglxxii/src/PacManV8/docs/field-manual/vanguard8-build-directory-skew-and-timed-opcodes.md`

Allowed scope:
- Timed opcode coverage inside the existing extracted HD64180 runtime for
  the six register-pair stack opcodes listed above only
- Tests and docs needed to lock the newly covered timed opcode surface
- Narrow runtime/integration coverage for the observed PacManV8 VBlank
  handler prologue/epilogue path

Explicit non-goals:
- Full Z80/Z180 opcode completion (no broad stack-family sweep beyond
  register-pair stack opcodes; no IX/IY/index-register stack forms; no
  stack-pointer arithmetic)
- Any additional PUSH/POP beyond the six register-pair opcodes named above
- New index-register, CB-prefixed, or block-instruction work
- VDP, renderer, scheduler, or audio changes unrelated to the timed CPU
  blockers
- Frontend or ROM-side work bundled into the same milestone
- Opening a broader "general opcode audit" effort inside this milestone;
  any additional gaps surfaced by the full PacManV8 run past the VBlank
  handler belong to a future narrow compatibility milestone

Allowed paths:
- `src/core/`
- `third_party/z180/`
- `tests/`
- `docs/`

Required tests:
- Focused timed CPU coverage for `PUSH BC` (`0xC5`), `PUSH DE` (`0xD5`),
  and `PUSH HL` (`0xE5`) that pins the documented semantics: SP is
  predecremented by two, the high byte of the named register pair is
  written to (SP+1) and the low byte to SP, PC advances by one, 11
  T-states, flags untouched
- Focused timed CPU coverage for `POP BC` (`0xC1`), `POP DE` (`0xD1`),
  and `POP HL` (`0xE1`) that pins the documented semantics: the low byte
  of the named register pair is read from SP and the high byte from
  (SP+1), SP is post-incremented by two, PC advances by one, 10
  T-states, flags untouched
- Regression coverage that the already-registered `PUSH AF` / `POP AF`
  opcodes remain covered in the timed path after the new opcodes are
  added
- Runtime/integration coverage reproducing the PacManV8 VBlank handler
  prologue (`PUSH AF; PUSH BC; PUSH DE; PUSH HL`) and epilogue
  (`POP HL; POP DE; POP BC; POP AF`) around the `audio_update_frame`
  call and proving timed execution survives all six previously missing
  opcode PCs

Exit criteria:
- The emulator no longer aborts on the documented PacManV8 VBlank
  handler path because timed execution is missing any of `0xC5`,
  `0xD5`, `0xE5`, `0xC1`, `0xD1`, or `0xE1`
- The added timed opcode coverage is narrow, documented, and
  regression-covered
- Remaining opcode gaps (beyond the six designated by the field manual)
  stay explicit and deferred, rather than expanding this milestone
  into a general CPU-completion pass
- The PacManV8 T016 regression command against the canonical
  `cmake-build-debug` Vanguard8 headless binary clears the VBlank
  handler prologue and epilogue without a timed-opcode abort
  (final T016 verification happens in the PacManV8 repo, not here)

Forbidden extras:
- Turning the milestone into a broad CPU rewrite or generic stack-family
  table (no IX/IY PUSH/POP forms, no `EX (SP),HL`, no `LD SP,rr`)
- Bundling unrelated real-ROM compatibility fixes under the same task
- Loosening existing timed CPU or integration tests instead of adding
  the missing opcode coverage
- Adding VDP, palette, audio, or frontend changes to this milestone
- Treating the already-removed stale `build/` CMake directory as an
  authoritative binary location; the canonical path is
  `cmake-build-debug/src/vanguard8_headless`

Verification commands:
```bash
cmake --build cmake-build-debug
ctest --test-dir cmake-build-debug --output-on-failure
```
