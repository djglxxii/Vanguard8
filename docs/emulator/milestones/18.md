# Milestone 18 — Headless Runtime Frame-Dump Parity

Objective:
- Make the headless frame-dump path reflect actual runtime output so mixed-mode
  scenes can be inspected reliably outside the desktop frontend.

Required docs:
- `docs/spec/02-video.md`
- `docs/emulator/04-video.md`
- `docs/emulator/07-implementation-plan.md`

Allowed scope:
- Headless/runtime frame-dump behavior under `src/frontend/`
- Tests and docs needed to lock parity between dumped frames and compositor
  output
- The smallest CLI/documentation adjustment needed to distinguish runtime dumps
  from fixture-only behavior

Explicit non-goals:
- General frontend/window-host refactoring
- New capture formats, video encoders, or debugger media tooling
- Renderer changes unrelated to dump correctness
- Showcase-ROM scene changes

Allowed paths:
- `src/frontend/`
- `tests/`
- `docs/`

Required tests:
- Headless regression coverage proving runtime dumps come from the active ROM
  session
- Coverage proving dumped frame dimensions match the compositor width for both
  `256x212` and `512x212` output
- A narrow parity check between dumped frame bytes and the runtime compositor
  frame used for hashing

Exit criteria:
- `--dump-frame` on a running ROM produces a trustworthy runtime inspection
  artifact
- Mixed-width scenes dump at `512x212` when the compositor resolves to the
  wider grid
- Any remaining fixture-only capture path is explicit and documented

Forbidden extras:
- Broad CLI redesign or debugger/capture feature expansion
- Unrelated ROM or core-emulation changes bundled into the same milestone

Verification commands:
```bash
cmake -B build -DCMAKE_TOOLCHAIN_FILE=third_party/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build
ctest --test-dir build --output-on-failure
```
